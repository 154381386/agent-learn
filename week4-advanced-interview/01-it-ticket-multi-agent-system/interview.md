# 4.1 IT 工单机器人：多 Agent 系统架构设计 — Interview

## 高频面试题
1. 你设计的 IT 工单机器人，为什么用户侧只保留一个机器人角色？
2. 罗伯特作为编排层，核心职责有哪些？
3. Agent、Skill、MCP 三者分别是什么关系？
4. 为什么要先做 RAG，再决定是否进入 Agent 诊断？
5. 为什么要采用中心化编排，而不是让多个 Agent 自由通信？
6. 子 Agent 为什么要接收结构化任务包，而不是整段聊天记录？
7. 高危操作如何做到“安全控制在代码层而不是 prompt 层”？
8. 串行诊断和并行诊断分别适合什么场景？
9. 多个 Agent 结论冲突时怎么处理？
10. Planning 和并行诊断是什么关系？
11. 主动预诊断系统为什么一定要先做告警聚合？
12. 工单结束后，系统如何形成闭环学习？
13. 多 Agent 系统最关键的观测指标有哪些？
14. 这个项目你实际用了什么技术栈，为什么这么选？
15. 你怎么证明这不是一个 Prompt Demo，而是工程化 Agent 系统？
16. 这个系统为什么要按 V1 → V5 逐步演进？
17. 为什么执行失败后先转人工，而不是直接自动恢复？
18. 如果后面要做更细的自动恢复，应该先补什么？
19. 历史案例召回应该在什么阶段触发，为什么后来拆成“自动 hint + 显式 tool”？

## 面试答题要点
- **单前台角色**：降低用户心智负担，统一语气、权限、审计与上下文管理。
- **罗伯特职责**：收消息、做路由、先 RAG、再调 Agent、控审批、汇总回复、沉淀知识。
- **三层关系**：Agent 是会推理的角色，Skill 是被动工具，MCP 是工具接入协议。
- **RAG 优先**：大量工单是知识问答，没必要一上来就跑实时诊断，先降低延迟和成本。
- **中心化编排**：更可控、好审计、易排障，尤其适合企业生产环境。
- **结构化任务包**：降噪、可审计、方便重放，也便于替换模型和演进 Agent。
- **代码层安全**：动作分级、参数校验、审批网关、受控执行器，不能只靠 prompt 约束。
- **并行策略**：路由置信度高走串行，置信度低走 fan-out 并行，用时间换准确率。
- **冲突处理**：不硬选，显式展示分歧、降低置信度，必要时转人工。
- **Planning vs 并行**：Planning 是战略层；并行诊断是单步执行时的战术层优化。
- **主动预诊断**：先聚合告警，避免告警风暴、重复诊断和刷屏；再做价值评估和主动通知。
- **闭环学习**：把案例、用户画像、统计指标持续沉淀，让系统越用越聪明。
- **关键指标**：自动解决率、转人工率、Agent 成功率、RAG 命中率、审批通过率、成本归因。
- **技术栈口径**：`Python + FastAPI + LangGraph + PostgreSQL/pgvector + Redis`，兼顾 LLM 编排效率和工程可维护性。
- **工程化证明**：消息幂等、状态机编排、工具权限网关、审批回调、审计日志、可观测性。
- **渐进演进**：先做 MVP 证明价值，再逐步加 Agent、闭环学习、复杂诊断、主动巡检。
- **失败先转人工**：执行层失败时，系统先暴露 checkpoint、failed step、operator hints，不贸然自动重试，避免非幂等副作用扩大。
- **自动恢复前提**：必须先补动作幂等性、外部状态确认、白名单策略和独立评估，再谈自动化恢复。
- **案例召回分层**：模糊首问不适合直接强召回历史案例，否则容易把诊断带偏；现在拆成“query 足够具体时做自动弱 hint”加“拿到 live evidence 后再显式查案例 tool”。

## 一分钟回答模板
> 我会把这个系统设计成一个中心化编排的多 Agent 架构。用户侧只看到一个飞书机器人“罗伯特”，它负责理解问题、先走 RAG 检索、再按需调度专业 Agent 做实时诊断，并统一做风险审批和结果汇总。后台的 Pod 分析、根因诊断、网络诊断等 Agent 不直接和用户对话，只接收结构化任务包。对于高置信度问题走串行诊断，对模糊问题走并行 fan-out，再做置信度融合。高危操作必须通过代码层审批网关执行。工单结束后再把根因、证据、处理动作沉淀到知识库和记忆系统，形成闭环学习；进一步还能接入告警系统做主动预诊断，逐步演进成一个数字化 SRE 助手。

## 技术栈追问模板
> 我这套系统主要用 `Python + FastAPI + LangGraph` 来实现。`FastAPI` 负责飞书 webhook、审批回调和 Agent HTTP 接口；`LangGraph` 负责把 RAG、路由、并行诊断、审批等待、总结入库这些步骤串成一个有状态工作流；存储层用 `PostgreSQL + pgvector` 管知识库和工单案例，用 `Redis` 管短期上下文、幂等键和缓存。之所以这么选，是因为 Python 的 LLM 生态最成熟，这套栈原型开发快，后续拆服务和做观测也比较顺手。

## 工程化追问模板
> 我会特别强调这不是一个只靠 prompt 的 demo，而是做了四层工程化：第一层是飞书接入和消息幂等，防止回调重复；第二层是 orchestrator 状态机，把一张工单从提问、诊断、审批到总结串起来；第三层是工具网关，所有 K8s 和监控操作都要经过权限校验和参数校验；第四层是审计和可观测性，所有 Agent 调用、审批动作和最终执行都会落库，方便复盘和追踪。

## 执行失败策略
> 当前实现里，执行失败后不会直接自动恢复，而是统一转人工介入。原因很实际：审批通过不代表外部动作一定可安全重放，很多动作可能已经部分生效，系统如果不知道资源真实状态，自动 retry 反而会把问题扩大。所以现在系统做的是把 checkpoint、failed step、recovery hints 都保留下来，帮助值班人快速接手，而不是盲目续跑。

## 自动恢复演进模板
> 如果后面要做更细的自动恢复，我不会先开一个通用 retry API，而是先补四件事：第一，给动作定义幂等性和 operation_id；第二，能区分动作未执行、已执行、状态不确定三种外部状态；第三，只对白名单幂等动作开放自动恢复；第四，单独做恢复评估集，验证误恢复率。这样自动化才是可控的。

## 历史案例召回演进模板
> 这块我一开始做得比较直觉化，只要进入诊断就先按用户当前问题查一轮历史案例，想把它当背景知识补充进去。后来发现这个设计在首问很模糊时准确率不高，反而会把 supervisor 往错误方向带。于是我把它拆成两层：第一层是自动预召回，但只在 query 已经足够具体时才触发，而且它只作为 hint，不当成现场证据；第二层是显式 `search_similar_incidents` tool，让模型先做 1 到 2 个只读检查，拿到更具体的 live evidence 后再主动查历史案例。这样既保留了历史工单复用价值，又避免过早先验污染当前诊断。

## 已实现 vs 规划中
- **已实现**：飞书机器人接入、RAG 检索、中心化 orchestrator、2-3 个核心诊断 Agent、审批卡片回调、案例总结入库。
- **已实现**：历史案例召回双层策略，包含“首跳弱 hint 预召回 + 诊断中显式案例搜索 tool”。
- **规划中**：主动预诊断、更细粒度的恢复白名单、系统化离线评估和用户画像增强。
- **回答原则**：只把自己真正跑通过的链路说成“已实现”，把架构图里的增强项明确说成 roadmap。
