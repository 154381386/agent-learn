# 4.1 IT 工单机器人：多 Agent 系统架构设计 — Notes

## 本节定位
- 这是一个适合面试展开讲 10-15 分钟的完整系统设计案例。
- 主题是：**企业内部 IT 工单的自动诊断、自动处理、人工协同与闭环学习**。
- 目标不是“造一个会聊天的机器人”，而是设计一个**可控、可审计、可演进**的多 Agent 生产系统。

## 一句话版本
> 用户只和一个叫“罗伯特”的飞书机器人对话；罗伯特负责理解问题、检索知识、调度多个专业 Agent、控制风险审批、汇总结果，并在工单结束后沉淀经验，让系统越用越聪明。

## 一、业务场景与设计原则

### 业务场景
企业内部 IT 工单群里，用户会问：
- `order-service Pod 频繁重启`
- `支付接口超时`
- `昨天发版后 CPU 飙高`
- `为什么这个服务这个月云成本翻倍`

系统需要做到：
- 能直接回答的，先基于知识库和历史工单快速回复。
- 需要深入诊断的，自动调度多个专业 Agent 收集证据。
- 涉及高风险动作的，必须人工审批。
- 解决完成后，自动把这次工单转成未来可复用的知识。

### 核心设计原则
```text
能自动解决的不打扰人
需要人介入的先把功课做好
高危操作必须人工审批
安全控制在代码层而非 prompt 层
每次工单都让系统变得更聪明
```

## 二、系统总览

### 用户视角：只有一个机器人
用户侧永远只看到“罗伯特”，不会直接看到子 Agent。

```text
┌─────────────────────────────────────────────┐
│ 工单群                                       │
│                                             │
│ 张三：order-service Pod 频繁重启              │
│                                             │
│ 罗伯特🤖：正在诊断中...                       │
│ 罗伯特🤖：诊断结果：OOM Killed，建议回滚       │
│ 罗伯特🤖：⚠️ 高危操作 [批准] [拒绝]           │
│                                             │
│ 李四(值班)：[批准]                            │
│                                             │
│ 罗伯特🤖：✅ 回滚完成，服务已恢复              │
└─────────────────────────────────────────────┘
```

### 内部视角：中心化编排
```text
飞书消息 → 罗伯特（编排层）
        → RAG 检索（历史工单 / SOP / 用户画像）
        → 专业 Agent 诊断（Pod / 根因 / 网络 / 云成本）
        → 审批网关（高危动作）
        → 回复用户 / 升级值班人
        → 工单总结与记忆入库
```

### 为什么只保留一个前台角色
- 降低用户心智负担，用户永远只找一个机器人。
- 避免多个 Agent 在群里各说各话，体验混乱。
- 便于统一语气、统一审计、统一权限控制。
- 全局上下文只保留一份，减少多 Agent 间上下文漂移。

## 三、罗伯特：唯一对外编排者

### 罗伯特的职责
1. 接收飞书群消息与事件。
2. 做意图识别、复杂度判断、用户画像检索。
3. 优先走 RAG，尽量直接答。
4. 需要深入诊断时，把任务派发给对应子 Agent。
5. 维护全局上下文，并把**裁剪后的结构化上下文**传给子 Agent。
6. 识别是否涉及高危动作，必要时发审批卡片。
7. 汇总最终结论，用统一口径回复群里用户。
8. 在工单结束后做总结、沉淀知识、更新用户画像。

### 为什么要中心化编排
```text
罗伯特 → Pod 分析 Agent → 得到 Pod 状态与日志证据
罗伯特 → 根因诊断 Agent → 结合变更历史做归因
罗伯特 → 网络诊断 Agent → 排除网络因素
罗伯特 → 汇总并回复用户
```

优点：
- 流程可控，容易审计。
- 全局上下文集中，决策质量更稳定。
- 易定位问题：是路由错了，还是某个 Agent 结果差。
- 更容易做 SLA、超时、重试、熔断与成本归因。

## 四、三层模型：Agent vs Skill vs MCP

### 分层关系
```text
┌─────────────────────────────────┐
│ Agent 层（有大脑，会推理）       │
│ 罗伯特、Pod 分析、根因诊断...    │
└──────────┬──────────────────────┘
           │ 调用
           ↓
┌─────────────────────────────────┐
│ Skill 层（无大脑，被动执行）     │
│ 查 Pod、查日志、查监控、查变更    │
└──────────┬──────────────────────┘
           │ 标准化接入
           ↓
┌─────────────────────────────────┐
│ MCP 层（工具接入协议）           │
│ K8s MCP / Prometheus MCP ...   │
└─────────────────────────────────┘
```

### 核心区别
| 概念 | 是什么 | 类比 |
|------|--------|------|
| Skill | 被动工具，执行固定流程 | 锤子、螺丝刀 |
| Agent | 有角色与推理能力 | 工程师 |
| MCP | 工具接入标准协议 | USB 接口 |

### 面试时的关键表述
- **Agent 不是 MCP。**
- **MCP 也不是智能体。**
- 正确关系是：**Agent 通过 Skill 完成任务，而 Skill 可以通过 MCP 标准化接入外部系统。**

## 五、诊断流程：三阶段闭环

### 第一阶段：先 RAG，快速直答
罗伯特收到问题后一定先尝试检索：
- 历史工单库：之前同类问题怎么解决的
- 运维知识库：SOP、Runbook、FAQ、架构文档
- 用户画像：用户是否是研发、运维、产品，技术背景如何

价值：
- 一部分问题根本不需要进入 Agent 深度诊断。
- 能显著降低延迟和 token 成本。
- 回答风格可以根据用户画像自适应：对新手讲结论，对专家讲证据。

### 第二阶段：子 Agent 自动诊断
当满足以下任一条件时，进入 Agent 诊断：
- RAG 命中率低
- 用户明确说“继续排查”
- 问题本身需要实时环境信息
- 需要多源证据才能归因

典型专业 Agent：
- `Pod 分析 Agent`：重启、OOM、探针失败、CrashLoop
- `根因诊断 Agent`：结合最近变更、日志、依赖与时间线做归因
- `网络诊断 Agent`：DNS、连通性、时延、丢包
- `云成本 Agent`：资源浪费、账单异常、成本归因

### 第三阶段：人机共创
触发条件：
- 用户主动要求拉值班人
- 操作风险高，需要审批
- 多 Agent 结论冲突
- 超出系统能力边界

此时罗伯特不只是“喊人”，而是先把功课做好：
- 当前症状
- 已采集证据
- 已排除方向
- 推荐操作及风险等级
- 需要值班人决策的关键点

这样值班人接手时不是从零开始，而是接一个**高质量诊断包**。

## 六、Agent 注册与任务协议

### Agent 注册表
编排层维护一份注册表，描述每个 Agent 的能力。

```json
{
  "agents": [
    {
      "name": "pod-analysis",
      "endpoint": "https://infra-team/agent/api/v1",
      "description": "诊断 Pod 状态异常、重启、OOM 等问题",
      "keywords": ["Pod", "重启", "OOM", "CrashLoop", "容器"],
      "input_schema": ["service", "namespace", "cluster", "symptom"],
      "timeout_sec": 10,
      "risk_actions": ["rollout_restart", "rollback"]
    }
  ]
}
```

### 为什么一定要传结构化任务包
不要把整段聊天原样扔给子 Agent，而是传：
- 工单 ID
- 用户问题摘要
- 当前上下文摘要
- 已知事实
- 待验证假设
- 可用工具范围
- 时间预算 / token 预算

这样做的好处：
- 降低上下文噪音
- 便于审计与重放
- 子 Agent 可以做得更专更稳
- 方便后续替换模型或重写某个 Agent

### 示例任务包
```json
{
  "ticket_id": "INC-20260312-001",
  "service": "order-service",
  "cluster": "prod-shanghai-1",
  "symptom": "Pod 频繁重启",
  "known_facts": [
    "过去 10 分钟重启 6 次",
    "最近 2 小时发布过 v2.3.1"
  ],
  "questions": [
    "是否 OOM 或探针失败？",
    "是否与最近变更相关？"
  ],
  "constraints": {
    "timeout_sec": 8,
    "allowed_tools": ["k8s.describe_pod", "k8s.logs", "monitor.query"]
  }
}
```

## 七、工程落地：技术栈与服务边界

### 推荐你在面试里采用的技术栈口径
如果面试官追问“你到底怎么实现的”，可以把系统拆成下面这套很真实、也很常见的 Python 技术栈：

| 层 | 技术选型 | 作用 |
|----|----------|------|
| 机器人接入层 | `FastAPI` + 飞书 Bot Webhook | 接收群消息、卡片回调、审批事件 |
| 编排层 | `LangGraph` + Python 自定义状态机 | 驱动 RAG、路由、并行诊断、审批等待、总结入库 |
| Agent 服务层 | `FastAPI` 微服务 / 统一 Agent Runtime | Pod 分析、根因诊断、网络诊断等专业 Agent |
| 模型层 | `OpenAI API` / 企业统一模型网关 | 意图识别、总结、诊断推理、结果融合 |
| 检索层 | `pgvector` + PostgreSQL | 历史工单、SOP、FAQ、用户画像向量检索 |
| 缓存 / 会话层 | `Redis` | 工单上下文缓存、幂等键、短期记忆、限流 |
| 异步任务层 | `Celery` 或 `asyncio` 后台任务 | 总结入库、异步预诊断、非阻塞告警处理 |
| 可观测性 | `Prometheus` + `Grafana` + `OpenTelemetry` | Agent 成功率、链路 Trace、耗时、成本统计 |
| 部署 | `Docker` + `Kubernetes` | 编排服务与子 Agent 的容器化部署 |

### 为什么这里推荐 `LangGraph`
- 因为这个系统不是一次性函数调用，而是**有状态的长流程**：RAG → 路由 → 子 Agent → 审批等待 → 继续执行 → 总结入库。
- `LangGraph` 很适合表达这种带分支、回环、人工中断恢复的流程。
- 如果面试官问为什么不用纯 `LangChain`，你可以答：`LangChain` 更像组件库，而 `LangGraph` 更适合编排有状态工作流。

### 服务拆分建议
```text
1. feishu-gateway
   - 接收飞书消息和审批回调
   - 做签名校验、幂等、鉴权

2. robert-orchestrator
   - 维护工单状态机
   - 负责 RAG、路由、上下文裁剪、汇总回复

3. agent-runtime
   - 统一封装 Agent 调用协议
   - 管理超时、重试、并行 fan-out/fan-in

4. tool-gateway
   - 封装 K8s、监控、变更系统、CMDB 等工具
   - 统一权限校验、参数校验、审计

5. memory-service
   - 管理历史工单、知识库、用户画像、总结入库
```

### 为什么不用“一个大服务全做完”
- 机器人接入、编排、工具执行、记忆管理的稳定性诉求不同。
- 工具网关必须更保守，权限控制要独立。
- Agent 迭代频繁，而接入层和审批链路要求更稳定，拆开后更容易灰度发布。

### 一条真实的请求链路
```text
飞书群消息
  → feishu-gateway 校验签名，落一条 ticket_event
  → robert-orchestrator 创建 TicketState
  → 先调 retriever 查 SOP / 历史工单
  → 命中低则走 router 决定串行还是并行
  → agent-runtime 调 pod-analysis / rca / network
  → 如需回滚，则生成 approval_request
  → 值班人点击审批卡片
  → orchestrator 恢复状态机，调用 tool-gateway 执行
  → 结果回群 + 生成 summary + 写入 case memory
```

### 数据存储怎么分
- `PostgreSQL`：工单主表、审批记录、审计日志、案例库元数据。
- `pgvector`：历史工单、SOP、FAQ 的向量索引。
- `Redis`：工单上下文缓存、会话短期记忆、消息去重、锁。
- 对象存储：长日志、诊断报告快照、卡片附件。

### 关键表设计
```sql
ticket(id, source, status, severity, service, assignee, created_at)
ticket_message(id, ticket_id, role, content, created_at)
agent_run(id, ticket_id, agent_name, input_json, output_json, latency_ms, status)
approval_request(id, ticket_id, action, risk_level, params_json, status, approver)
knowledge_case(id, service, symptom, root_cause, resolution, embedding, review_score)
user_profile(user_id, team, skill_level, answer_preference, updated_at)
audit_log(id, ticket_id, actor, action, target, result, created_at)
```

### RAG 的实现细节
- 文档切分：按 `SOP 步骤 / 故障现象 / 根因 / 处理动作` 语义切块，而不是机械按字数切。
- 检索策略：`BM25 + 向量检索` 混合召回，避免纯向量检索漏掉精确术语。
- 重排：先召回 20 条，再让轻量 reranker 或 LLM 重排前 5 条。
- 引用策略：回答时优先返回“结论 + 证据来源 + 是否建议继续诊断”。

### 路由策略怎么做得更可信
不要说“完全靠大模型感觉判断”，那样会显得虚。更可信的说法是：
- 第一层用规则快速识别明显模式，如 `OOM`、`CrashLoopBackOff`、`DNS`。
- 第二层再用 LLM 做模糊意图分类与复杂度评分。
- 最终输出结构化路由结果：

```json
{
  "intent": "service_unavailable",
  "confidence": 0.45,
  "complexity_score": 0.76,
  "recommended_mode": "parallel",
  "candidate_agents": ["pod-analysis", "root-cause", "network-diagnosis"]
}
```

### 并行诊断的工程实现
- 编排层使用 `asyncio.gather` 或 `TaskGroup` 并行调多个 Agent。
- 每个 Agent 调用带独立超时与重试策略。
- 汇聚层统一收集 `conclusion / evidence / confidence / suggested_actions`。
- 超时的 Agent 不阻塞整体结论，但会进入 Trace 方便排障。

### 审批链路怎么说更像真实项目
- 飞书卡片里只放“动作摘要 + 风险等级 + 参数预览 + 批准/拒绝”。
- 卡片点击后回调到 `feishu-gateway`，写入 `approval_request` 状态。
- `orchestrator` 监听到审批通过后才恢复工作流。
- 真正执行动作的是 `tool-gateway`，它再次校验环境、参数、审批状态和操作者身份。

### 你可以明确说“已实现”和“规划中”
面试时最怕把所有蓝图都说成已经上线。更可信的讲法是分层表达：

**已实现 / 已验证过的部分**
- 飞书机器人接入、消息收发、审批卡片回调。
- 一个中心化 orchestrator，能维护单工单状态。
- RAG 检索 + 2 到 3 个核心 Agent（例如 Pod / 根因 / 网络）。
- 高危操作审批网关与审计日志。
- 工单总结后写回案例库。

**正在完善 / 下一阶段**
- 主动预诊断全链路。
- 更细粒度的用户画像自适应回复。
- 更系统的离线评估与 A/B 路由实验。
- 自动化回滚的更严格保护机制。

这样的说法会让面试官觉得你不是在“吹一个全能架构图”，而是真正做过分阶段落地。

## 八、权限与审批：安全在代码层，不在 Prompt 层

### 核心原则
模型可以“建议”，但不能直接获得危险权限。

### 具体做法
- Skill 按风险等级划分：`read` / `write-low-risk` / `write-high-risk`
- 只有读操作可默认自动执行。
- 高风险动作必须经过**代码里的审批网关**。
- Agent 返回的是“意图 + 参数”，真正执行由受控执行器完成。

### 风险分级示例
| 动作 | 风险级别 | 处理方式 |
|------|----------|----------|
| 查看 Pod 状态 | low | 自动执行 |
| 查询近 30 分钟日志 | low | 自动执行 |
| rollout restart | medium | 可配置审批 |
| 生产回滚 | high | 必须人工审批 |
| 删除资源 | critical | 默认禁用 |

### 为什么说安全控制必须在代码层
因为 prompt 只是“建议模型守规矩”，不是强制约束。
真正可靠的方式是：
- 工具侧鉴权
- 参数校验
- 风险分级
- 审批工作流
- 完整审计日志

## 九、并行诊断与置信度融合

### 问题：串行诊断太慢
如果用户只说：`服务挂了，不知道怎么回事`，系统并不确定该优先查 Pod、网络、变更还是依赖。此时纯串行会导致等待太久。

### 路由策略：分级路由 + 条件并行
```text
意图明确（confidence > 0.8）：
  直接串行调用最相关 Agent
  优点：快、省 token

意图模糊（confidence < 0.8）：
  并行 fan-out 给多个候选 Agent
  优点：用时间换准确率
```

### 并行调度示意
```text
用户："order-service 挂了"
    ↓
罗伯特意图识别：confidence = 0.45
    ↓
并行派发
    ├─ Pod 分析 Agent   → 5s 返回
    ├─ 根因诊断 Agent   → 3s 返回
    └─ 网络诊断 Agent   → 8s 超时
    ↓
结果汇聚 + 置信度融合
```

### 超时策略
```python
PARALLEL_CONFIG = {
    "hard_timeout": 15,
    "soft_timeout": 8,
    "min_results": 1,
}
```

设计思路：
- `hard_timeout`：最多等多久，防止无限等待。
- `soft_timeout`：已有足够结果就提前汇聚。
- `min_results`：至少拿到 1 个结果再做结论。

### 置信度融合规则
1. **结论一致**：多个 Agent 互相印证，提升置信度。
2. **结论互补**：合并成更完整的诊断链路。
3. **结论冲突**：不强行拍板，显式降置信度并拉人。

### 融合输出示例
```json
{
  "fused_diagnosis": {
    "conclusion": "v2.3.1 引入内存泄漏导致 OOM Killed",
    "confidence": 0.92,
    "sources": [
      {
        "agent": "Pod分析",
        "said": "OOM Killed，内存峰值 800Mi / 限制 512Mi",
        "confidence": 0.85
      },
      {
        "agent": "根因诊断",
        "said": "2 小时前发版 v2.3.1，变更涉及数据库连接池",
        "confidence": 0.72
      },
      {
        "agent": "网络诊断",
        "said": "网络连通性正常，排除网络因素",
        "confidence": 0.95
      }
    ],
    "fusion_logic": "OOM 证据与最近变更互相印证，同时网络方向被排除",
    "suggested_actions": [
      {
        "action": "回滚到 v2.3.0",
        "risk": "high",
        "reason": "多个 Agent 支持该方案"
      }
    ]
  }
}
```

### 这里体现的系统观
- 系统不应该在不确定时假装确定。
- “不知道”也是一种高质量输出。
- 多 Agent 的价值不是“票数多者获胜”，而是**交叉验证与证据互补**。

## 十、ReAct 与 Planning：双模式协同

### 为什么不能所有工单都走 Planning
Planning 很强，但更慢、更贵，也更容易过拟合复杂流程。
简单工单如果也强制做计划，会把本来 1 步解决的问题拖成 5 步。

### 双模式策略
```text
简单工单 → ReAct
复杂工单 → Planning
```

### 复杂度评分建议
满足 2 条及以上，就从 ReAct 升级为 Planning：
- 涉及多个服务
- 症状模糊
- 没有历史相似案例
- 出现“分析一下 / 排查看看 / 可能不是单点问题”等探索性表达
- 预计需要多轮工具调用

### Planning 的职责：战略层
Planning 负责决定：
- 整体要分几步
- 先查什么，再查什么
- 哪一步需要并行
- 什么情况下 replan

### 并行诊断的职责：战术层
Planning 决定“要深入诊断异常服务”；至于这一步内部是并行调 Pod/网络/根因 Agent，属于战术层。

**一句话：Planning 决定做什么，并行诊断决定这一步怎么更快做完。**

### Replan 触发条件
只有出现关键信号才 replan，而不是每一步都重排：
- 新异常服务被发现，范围扩大
- 发现共同依赖，诊断方向改变
- 某一步执行失败
- 新证据与旧结论冲突

### 硬性护栏
```text
最多 6 步
最多 replan 2 次
总超时 5 分钟
超过上限直接拉人
```

## 十一、主动预诊断系统

### 设计目标
从“用户来报障”升级为“系统先发现异常并给出预诊断”。

### 整体流程
```text
Prometheus / K8s Events / 发布事件
          ↓
告警接入层
（去重、关联、严重度评估）
          ↓
是否值得预诊断
    ├─ 否：记录日志，不打扰用户
    └─ 是：进入预诊断 Pipeline
              ↓
        查历史工单 / 最近变更 / 当前指标 / 环境状态
              ↓
        LLM 综合研判
              ↓
        生成预诊断报告并主动通知服务负责人
```

### 为什么要先做聚合再做 LLM
一次故障可能同时触发：
- Pod 重启告警
- 错误率升高告警
- 时延上升告警
- K8s Event 异常

如果每条告警都独立跑一次诊断：
- 成本高
- 容易刷屏
- 还会让多个诊断彼此重复

### 告警聚合器设计
```python
class AlertAggregator:
    def __init__(self):
        self.buffer = {}      # dedupe_key -> recent alerts
        self.incidents = {}   # incident_id -> aggregated incident

    def make_dedupe_key(self, alert):
        return (
            alert["service"],
            alert["cluster"],
            alert["alert_type"],
            alert["severity"]
        )

    def ingest(self, alert):
        key = self.make_dedupe_key(alert)
        self.buffer.setdefault(key, []).append(alert)

        incident = self.merge_if_related(key)
        return incident

    def merge_if_related(self, key):
        # 时间窗口内、同服务、同集群、同严重度的告警归并为一个 incident
        return {
            "service": key[0],
            "cluster": key[1],
            "incident_type": key[2],
            "severity": key[3],
            "status": "aggregated"
        }
```

### 值不值得打扰用户
不是每个 incident 都值得主动通知。可以做一层价值评估：
- 是否是低优先级告警
- 是否是已知问题且已静默
- 是否刚刚已经通知过同一负责人
- 是否有足够高的业务影响

### 主动通知示例
```text
@order-service 负责人
检测到 order-service 在 prod-shanghai-1 出现异常：
- 10 分钟内 Pod 重启 6 次
- 最近 2 小时发布了 v2.3.1
- 当前错误率上升 12%

预诊断结论：疑似新版本引入内存泄漏导致 OOM。
建议：优先检查 v2.3.1 相关变更；如影响扩大，建议审批回滚。
```

### 与被动诊断如何衔接
如果用户随后在群里继续提问：
- 不要从零开始再查一遍。
- 直接把该 incident 的预诊断结果作为上下文复用。
- 进入“预诊断 → 工单协同处理”的同一条链路。

## 十二、学习与记忆：让系统越用越聪明

### 工单结束后的三类沉淀
1. **案例记忆**：这次问题、证据、根因、处理动作、结果。
2. **用户画像**：用户更偏好结论式回答还是步骤式回答。
3. **统计指标**：哪个 Agent 命中率高，哪个场景最常见。

### 两种入库模式
- **无人值守解决**：LLM 自动总结后直接入库。
- **有人值守参与**：LLM 先生成总结，值班人 review / 打分后再入库。

### 为什么要有人 review
因为生产案例一旦入库，后续会被 RAG 检索反复使用。
如果把低质量结论直接入库，会形成“错误知识污染”。

### 推荐沉淀字段
```json
{
  "ticket_id": "INC-20260312-001",
  "service": "order-service",
  "symptom": "Pod 频繁重启",
  "root_cause": "v2.3.1 引入内存泄漏导致 OOM",
  "evidence": ["OOMKilled", "release timestamp matched"],
  "action": "rollback to v2.3.0",
  "outcome": "recovered",
  "review_score": 5
}
```

## 十三、可观测性与评估指标

### 为什么多 Agent 系统必须可观测
因为你要知道：
- 是路由有问题，还是某个 Agent 不稳定？
- 是 RAG 没命中，还是工具调用慢？
- 成本花在哪一层？

### 建议打的指标
- 工单总耗时
- RAG 命中率
- Agent 调用成功率 / 超时率
- 高风险操作审批通过率
- 自动解决率
- 转人工率
- 人工接手后的平均恢复时长
- 单工单 token / 成本归因

### 面试里最好准备 3 类“真指标”
如果项目真的做过，建议至少背住下面三组数据中的一组；没有真实数据也要准备统计口径：
- **效果类**：自动解决率、首次回答命中率、转人工率。
- **效率类**：平均响应时长、平均诊断时长、审批前准备时长。
- **稳定性类**：Agent 超时率、工具调用失败率、RAG 空召回率。

更可信的表达不是报一个特别漂亮的数字，而是说：
- 我们先统计 `baseline` 是人工处理平均 15 分钟。
- 机器人接入后，常见问题首响可以压到 10-30 秒。
- 真正需要人工的复杂 case，至少能把值班人接手前的证据准备好。

这种说法比“提升了 80% 效率”更像真实项目。

### Trace 粒度建议
每个工单保留完整链路：
- 用户原始问题
- 路由决策
- 调用了哪些 Agent
- 每个 Agent 看到了什么上下文
- 每一步用了哪些工具
- 哪一步触发了审批 / replan / 转人工
- 最终结论与置信度

## 十四、系统演进路径

```text
V1（MVP）
  罗伯特 + RAG 知识库 + 基础问答
  先解决 60% 的常见咨询问题

V2（加 Agent）
  增加 Pod 分析 / 根因诊断等专业 Agent
  接入 K8s、监控、变更系统

V3（闭环学习）
  工单自动总结 + review 入库
  引入用户画像与案例统计

V4（复杂工单能力）
  ReAct / Planning 双模式
  并行诊断 + 置信度融合
  全链路 Trace 与成本归因

V5（主动数字 SRE）
  告警接入、去重聚合、价值评估
  预诊断 + 主动通知 + 被动链路打通
```

### 为什么这样演进
因为真正的企业系统通常不是“一步到位”，而是：
- 先做能落地的 MVP
- 再逐步加自动化和闭环
- 最后才进入主动智能阶段

这比一开始就吹“全自动自治系统”更可信。

## 十五、面试里的高频追问答法

### Q1：为什么要中心化编排，而不是 Agent 彼此自由通信？
答法：
- 中心化更可控、可审计、好排障。
- 全局上下文集中在罗伯特，避免多个 Agent 之间上下文漂移。
- 对企业场景来说，稳定性和责任边界通常比“自治感”更重要。

### Q2：为什么先做 RAG，不直接多 Agent 开查？
答法：
- 因为很多工单本质上是知识问答，不需要实时诊断。
- 先 RAG 能降低延迟和成本，也符合“能自动解决的不打扰人”。

### Q3：两个 Agent 结论冲突怎么办？
答法：
- 不硬选，不装懂。
- 显式展示分歧、降低置信度，并把已收集证据交给值班人裁决。

### Q4：高危操作怎么防止误执行？
答法：
- 靠代码层权限网关，不靠 prompt 自觉。
- Agent 只能提出意图，不能直接获得危险执行权限。
- 高风险动作必须审批、鉴权、审计。

### Q5：并行诊断会不会浪费 token？
答法：
- 会，但只在低路由置信度场景触发。
- 在生产故障里，误诊成本和等待成本往往远高于 token 成本。

### Q6：Planning 和并行诊断是什么关系？
答法：
- Planning 是战略层，决定“做哪几步”。
- 并行诊断是战术层，决定“某一步怎么更快完成”。

### Q7：你这个项目的技术栈为什么这么选？
答法：
- 我用的是 `Python + FastAPI + LangGraph + PostgreSQL/pgvector + Redis` 这套栈。
- 原因是 Python 在 LLM 和工具编排生态最成熟，`FastAPI` 适合快速暴露 Agent API，`LangGraph` 适合有状态工作流，`PostgreSQL + pgvector` 能把业务数据和向量检索放在一套存储里，`Redis` 则负责短期状态和幂等控制。
- 这套方案的优点是原型开发快、后续可演进，也比较符合中小团队的维护成本。

### Q8：你怎么证明这不是一个“Prompt Demo”，而是工程系统？
答法：
- 我会强调我做了四层工程化：消息接入和幂等、编排状态机、工具权限网关、审计和可观测性。
- 尤其是审批链路和工具鉴权是在代码层做的，这一点和纯 prompt demo 区别很大。
- 同时我会区分哪些功能已经跑通，哪些还在 roadmap 上，避免把概念图讲成已落地系统。

## 十六、面试收束总结
可以用下面这段话收尾：

> 这个系统的核心不是堆很多 Agent，而是围绕企业 IT 工单场景，把“RAG 直答、专业 Agent 诊断、人工审批、闭环学习、主动预诊断”串成一个可控闭环。前台永远只有一个编排者，后台通过中心化调度多个 Agent，在低风险场景尽量自动化，在高风险场景严格审批，在不确定场景透明暴露分歧，最终让系统随着每次工单持续变聪明。

## 本节你需要记住的 8 句话
- 罗伯特是唯一对外角色，也是唯一全局上下文持有者。
- Agent 负责推理，Skill 负责执行，MCP 负责标准化接入。
- 默认先 RAG，再决定要不要进入 Agent 诊断。
- 技术栈可以落到 `Python + FastAPI + LangGraph + PostgreSQL/pgvector + Redis`。
- 路由最好是“规则优先 + LLM 补充”，而不是完全黑盒判断。
- 高危操作必须由代码层审批网关控制。
- 模糊问题走并行诊断，结果通过置信度融合汇总。
- 工单结束后要把经验沉淀回知识库和记忆系统。
- 面试时要明确区分“已实现”与“规划中”，这样最可信。
