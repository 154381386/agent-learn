这版设计针对你的场景：Supervisor + 多个 subagent + LangGraph + 中心化 HITL + 转人工。核心原则只有两条：

  - subagent 只负责分析、提案、执行建议，不直接各自 interrupt
  - 真正的 HITL 只在中心审批节点发生，执行和验证也由中心重新派发

  角色设计

  - supervisor
    职责：识别工单类型、路由 subagent、汇总结果、决定下一步
    输入：用户请求、当前工单状态、历史摘要
    输出：路由计划、下一步决策
    权限：不直接执行生产动作
  - k8s-agent
    职责：查 pod、deployment、event、rollout、配置挂载状态
    输入：服务、集群、命名空间、环境
    输出：k8s 事实、候选根因、候选动作
    权限：默认只读；高风险动作只能提案
  - cicd-agent
    职责：查发布记录、流水线状态、变更 diff、回滚候选
    输入：应用、环境、最近发布信息
    输出：发布事实、候选动作、风险标签
    权限：默认只读；回滚只能提案
  - observability-agent
    职责：查 metrics、logs、traces、alerts，负责执行后验证
    输入：服务、时间窗口、环境
    输出：影响范围、验证指标、恢复判断
    权限：只读
  - rca-judge-agent
    职责：证据冲突时做根因裁决和方案排序
    输入：多个 subagent 输出
    输出：根因排序、冲突说明、建议动作
    权限：不执行动作
  - approval-coordinator
    职责：收集 proposal、去重、合并、触发 interrupt、处理 resume
    输入：proposal 池、风险规则、审批状态
    输出：approved_actions / rejected_actions
    权限：发起审批，不执行生产动作
  - executor
    职责：执行已批准动作
    输入：approved_action、capability token
    输出：execution_result
    权限：只能通过 Execution Gateway
  - verifier
    职责：验证执行后效果
    输入：execution_result、verification_plan
    输出：verification_result
    权限：只读
  - handoff-agent
    职责：转人工、拉值班、发交接包
    输入：工单状态、证据、已尝试动作
    输出：handoff 事件
    权限：只操作 ticket/chat/paging

完整流程

  1. supervisor
     解析请求，决定要不要叫 k8s-agent、cicd-agent、observability-agent
  2. 各 subagent 并行分析
     只输出：
      - facts
      - evidence
      - hypotheses
      - candidate_actions
      - approval_proposals
  3. approval-coordinator
     读取所有 proposal，做：
      - 去重
      - 合并
      - 排序
      - 判断是否需要 HITL
  4. 如果不需要 HITL
     直接生成 approved_actions 进入执行
  5. 如果需要 HITL
     调 interrupt(payload)
  6. 用户审批后 resume(result)
     coordinator 把结果写入：
      - approved_actions
      - rejected_actions
  7. executor
     根据 approved_actions 执行
  8. verifier
     根据 verification_plan 做恢复验证
  9. supervisor
     根据验证结果判断：
      - done
      - 继续诊断
      - 再次审批
      - handoff

  HITL 的正确做法
  关键点：interrupt 只放在审批协调节点。

  错误做法：

  - k8s-agent 自己 interrupt
  - cicd-agent 自己 interrupt

  正确做法：

  - 两个 agent 只把 proposal 写进 approval_proposals
  - approval-coordinator 统一收口，调一次 interrupt

  approval coordinator 的逻辑
  它做 5 件事：

  1. collect
     收集所有 proposal
  2. dedupe
     同一 incident + 同一 resource + 同一 action 去重
  3. merge
     多个相近动作合并成 bundle
  4. policy check
     判断哪些动作真的需要审批
  5. interrupt
     如果需要人批，发一个统一审批 payload
6. 从 0 到 1 构建基于 LLM 的 AIOps 平台，实现告警智能分析、变更风险管理、故障自愈闭环，并确保 Human-In-The-Loop 安全管控。
7. Action:                                                                                                                                                                                                                                                                                                               
  - 基于 LangGraph + 自研 DeepAgents 框架 构建 SRE Agent，设计 8 层可组合 Middleware 栈（鉴权、审批、记忆、事件注入、错误恢复等），相比手写 LangGraph 减少 40-50% 样板代码                                                                                                                                              
  - 实现 Golden Signals 健康度评估引擎（时序异常检测 + 评分 + 洞察生成）和 Drain3 日志聚类，支撑自动化根因分析                                                                                                                                                                                                          
  - 设计两阶段 HITL 审批流（Authorization + Approval），确保高危操作有人兜底；结合 Evidence Chain 实现操作全链路审计                                                                                                                                                                                                    
  - 采用 CloudEvents 标准 构建事件驱动架构，支持多通道分发（IM 机器人、工单系统、EventBridge）                                                                                                                                                                                                                          
  - 落地 LangFuse + OpenTelemetry 双重可观测性，覆盖 LLM 调用追踪和分布式链路追踪                                                                                                                                                                                                                                       
                                                                                                                                                                                                                                                                                                                        
  Result: 平台已上线生产环境，覆盖告警分析、变更管理、运维请求三大核心场景，显著降低 MTTR。核心优势： 告警/部署场景下参数都是结构化的，LLM 不需要理解语义就能工作。

  ---
  痛点

  痛点 1：用户说话 Agent 听不懂

  用户: "华东 prod 的支付服务最近30分钟错误率高"

  Agent 需要自己从 prompt 规则里理解:
    "支付服务" → component=payment-service
    "华东"    → region=华东
    "prod"    → env=prod
    "30分钟"  → duration_minutes=30

  现实: prompt 有1500行，规则越多遵守率越低

  影响： 用户对话场景（operational_request）的准确率明显低于告警场景

  痛点 2：只能查"组件"，查不了其他类型

  用户: "订单库挂了影响哪些服务"

  Agent 的困境:
    - "订单库"不是一个 component 名字
    - 工具只接受 component 参数
    - 图里只有 component→component 的 calls 边
    - 没有 Database 类型的节点，无法反查"谁依赖这个库"

  影响： 跨类型排查（服务↔数据库、服务↔缓存、服务↔消息队列）做不了

  痛点 3：模糊描述无法定位

  用户: "订单相关的服务最近都有点慢"

  Agent 的困境:
    - "订单相关"匹配哪些组件？order-service? order-api? order-gateway?
    - 没有搜索能力，只能精确传名字
    - 要么猜一个，要么问用户 → 体验差

  影响： 用户必须知道精确的组件名才能用 Agent

  痛点 4：拓扑只有一层关系

  当前图里只有:
    component --calls--> component

  缺失的:
    component --queries--> database       ← Span 里的 db.name 没提取
    component --uses--> cache             ← Span 里的 net.peer.name 没提取
    component --publishes--> mq_topic     ← Span 里的 messaging.destination 没提取
    application --contains--> component   ← CMDB 里有但没接入

  影响： "支付服务依赖哪些中间件"回答不了，只能看到服务间调用

  痛点 5：排查靠 Prompt 硬编排，不灵活

  当前: system prompt 里写死了"先查指标→再查日志→再查拓扑"

  问题:
    - 所有告警都走同一个流程，不区分类型
    - 网络问题和代码 bug 的排查路径应该不同
    - 添加新排查逻辑 = 改 1500 行 prompt

  影响： 新场景全靠改 prompt，维护成本高

  解决方案

  方案 1：组件搜索工具（解决痛点 2、3）

  改动： 把已有的 ontology.py 包装成 Agent 可调用的工具

  # 新增一个工具注册到 ToolManager
  def search_components(keyword: str, env: str = None) -> list[dict]:
      """模糊搜索组件"""
      dsl = create_ontology_dsl(
          where_conditions={"name": [keyword]},  # 支持模糊匹配
      )
      result = await query_ontology_entity(request)
      return [{"name": e.name, "name_cn": e.name_cn, "owner": e.owner} for e in result]

  效果：
  用户: "订单相关的服务"
  Agent 调 search_components("order")
    → [order-service, order-api, order-gateway, order-db-proxy]
  Agent: 找到4个相关组件，我先查 order-service 的健康状况

  成本： 半天，改 manager.py 加一个工具

      # 匹配 entity_aliases → 提取 component 名

  效果：
  用户: "华东 prod 的支付服务"
  resolve_user_query → component=payment-service, env=prod, region=华东

  ---
  方案 3：从 Span 提取更多边（解决痛点 4）

  改动： 在 InsightFlow 的拓扑生成逻辑里，增加从 Span 属性提取跨类型边的规则

  现在只提取:
    span.service → span.peer.service  (component→component)

  增加:
    span.service + span.db.name      → component→database 边
    span.service + span.net.peer.name → component→cache 边（当 peer 是 Redis 时）
    span.service + span.messaging.destination → component→mq_topic 边

  效果：

  check_topology("payment-service")
  现在返回:  payment-service → order-service → user-service
  改进后返回: payment-service → order-service
              payment-service → order-db (Database)
              payment-service → payment-cache (Redis)
              payment-service → order-events (MQTopic)

  成本： 需要改 InsightFlow 侧，约 1-2 周。但这是收益最大的一步 — 打通了跨类型排查

  方案 4：类型 Catalog（解决痛点 5，为本体论打基础）

  改动： 加一个 object_type_catalog.json，让 Agent 知道"世界上有哪几种东西"

  {
    "Service": {
      "display_name": "微服务组件",
      "aliases": ["服务", "组件"],
      "tools": ["check_golden_signals", "check_logs", "check_traces"],
      "scope_fields": ["env", "region", "bz_cluster"]
    },
    "Database": {
      "display_name": "数据库",
      "aliases": ["库", "DB"],
      "tools": ["check_logs", "search_spans"],
      "scope_fields": ["env", "bz_cluster"]
    }
  }

  效果： Agent 能根据用户提到的对象类型，自动选择合适的工具组合，而不是对所有问题都跑同一套流程
