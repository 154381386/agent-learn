# Kubernetes 发布与 Deployment 使用手册

## 文档属性
- 文档类型：发布操作手册
- 适用对象：研发、发布值班、SRE
- 风险等级：高
- owner：发布平台组
- 关键词：`deployment`、`rollout`、`canary`、`rollback`

## 目标
规范服务发布、灰度、回滚与变更验证操作，避免研发通过临时手工方式绕开平台治理。

## 发布链路概览
```text
代码合并 → CI 构建镜像 → 生成版本号 → 预发验证 → 发布审批 → 生产灰度 → 生产全量 → 发布后观察
```

## 适用对象和职责
### 研发
- 负责代码变更与发布说明
- 负责确认是否存在 DB/配置兼容风险
- 负责 smoke test

### 值班 SRE
- 负责生产发布审批与风险评估
- 负责高危发布期间协同观察
- 负责异常时回滚与止血

### 发布平台
- 负责版本追踪、发布审计、灰度控制、回滚记录

## 发布前检查清单
### 代码层
- 是否合并了所有必须补丁
- 是否存在 Feature Flag 控制
- 是否变更了关键依赖版本
- 是否修改了启动参数、探针或资源配置

### 数据层
- 数据库 schema 是否向后兼容
- 是否存在异步任务/定时任务版本耦合
- 是否存在缓存 key 格式变更

### 运行层
- 是否有稳定的上一版本可回滚
- 是否有业务低峰窗口
- 是否有对应值班人在线
- 是否设置了发布观察面板

## Deployment 规范
- 每个服务必须显式配置 `resources.requests` 与 `resources.limits`
- 每个服务必须配置 `readinessProbe`
- 核心服务必须配置 `livenessProbe`
- `maxUnavailable` 默认 `25%`
- `maxSurge` 默认 `25%`
- 镜像 tag 必须是不可变版本，不使用 `latest`

## 发布平台标准流程
1. 选择目标服务和环境
2. 选择目标镜像版本，例如 `v2.3.1`
3. 填写发布单：变更摘要、风险点、回滚版本
4. 在 `staging-shanghai-1` 执行预发发布
5. 通过 smoke test 后申请生产发布
6. 生产环境先灰度 10%
7. 观察 5~10 分钟后再全量
8. 记录发布结果和观察结论

## 常用命令
```bash
kubectl -n default rollout status deploy/order-service
kubectl -n default rollout history deploy/order-service
kubectl -n default get rs -l app=order-service
kubectl -n default describe deploy order-service
```

## 灰度与放量建议
### 核心链路服务
- 先 10%
- 观察 5 分钟
- 再 30%
- 再 100%

### 非核心服务
- 可按 30% → 100% 简化

### 禁止直接全量的场景
- 涉及数据库连接池/线程池/缓存策略变更
- 涉及第三方依赖升级
- 历史上同服务在发布后容易出稳定性问题

## 发布后重点观察项
- 5xx 错误率
- P95 / P99 时延
- Pod 重启次数
- 内存曲线与 CPU 抖动
- 下游依赖错误率
- 消费堆积 / MQ lag

## 失败判断标准
满足任一条件，可进入回滚决策：
- 发布后 5 分钟内错误率 > 基线 + 5%
- 发布后 P95 时延 > 基线 + 50%
- 新 Pod 大量 `CrashLoopBackOff`
- 业务告警连续 3 分钟无恢复
- 新版本对关键路径产生明显副作用

## 回滚原则
- 先止血，后归因
- 回滚前确认上一版本仍可用
- 回滚后至少观察 10 分钟
- 回滚完成不代表根因解决，必须继续复盘

## 禁止事项
- 不允许直接 `kubectl edit deployment`
- 不允许线上临时改 ConfigMap 绕过发布流程
- 不允许没有 smoke test 就发生产
- 不允许没有回滚版本就上线破坏性变更

## 常见问题
### Q1：新版本只在一部分 Pod 出问题怎么办？
优先查看节点分布、探针配置、镜像拉取、配置加载差异，必要时暂停放量。

### Q2：预发没问题，生产有问题怎么办？
重点排查：生产流量、依赖规模、缓存数据、配置差异、DB 数据量差异。

### Q3：发布失败后是否一定立即回滚？
不一定。如果问题可快速降级且已控制影响，也可以先止血再评估。但核心链路一般优先回滚。

## 场景化案例

### 案例 1：payment-service 在 prod-shanghai-1 的现场处理
- 现象：用户反馈与 `payment-service` 相关的操作出现异常或体验下降。
- 初始信号：`P95 时延` 超过基线，值班人在 `grafana` 中看到异常趋势。
- 研判：先排除是否由 `错误回滚版本` 或最近变更导致。
- 建议动作：先做只读排查，再根据影响范围决定是否回滚、限流、降级或转人工审批。
- 记录要求：记录工单号、环境、服务名、时间窗、关键证据和处理结果。

- 说明：发布后 payment-service 错误率上升，首先确认最近 2 小时内是否存在版本、配置或依赖变更。
