# Mock Product Manuals

这是一组面向 `IT Ticket Agent` 的模拟企业内部知识库文档，目标不是写成“课堂笔记”，而是尽量贴近真实公司的平台手册、SOP、Runbook、FAQ 和值班文档。

这套文档主要用于三类场景：
- 作为 `IT Ticket Agent` 的初始 RAG 数据源，用来测试检索、重排、引用与答案融合能力。
- 作为面试或演示环境中的“企业内部知识库”素材，让系统看起来更接近真实生产场景。
- 作为后续离线评估样本的基础语料，用来测试长文档切分、混合检索、多文档引用与工单问答。

## 使用建议

### 1. 作为 RAG 数据源
推荐把 `mock_kb` 下所有文档按以下维度切块：
- 文档元信息：文档名、分类、关键词、适用对象、更新时间
- 操作步骤：适合做过程性问答
- 常见问题：适合 FAQ 检索
- 风险提示：适合高风险操作前置拦截
- 升级条件：适合转人工或审批判断
- 示例命令 / SQL / 配置片段：适合工具调用前的上下文补充

### 2. 作为测试集来源
可以基于这些文档设计如下测试：
- 单文档问答：如“堡垒机登录方式是什么？”
- 跨文档问答：如“Pod OOM 后是否应该回滚，回滚前还要看什么？”
- 模糊检索：如“服务挂了怎么查？”
- 带角色约束问答：如“研发能不能直接改生产数据库？”
- 风险识别：如“我要直接删 Pod 行不行？”
- 审批提示：如“生产回滚前需要谁来确认？”

### 3. 作为评测样本
建议从文档中抽取三类问题：
- 明确事实型：答案在一处文档中明确出现
- 半结构流程型：需要多段步骤拼装
- 规则判断型：需要结合风险说明和升级条件

## 推荐切分策略
### 适用于 FAQ 的切分
- 每个“常见问题”作为单独 chunk
- 附带上一层标题和文档标题

### 适用于操作流程的切分
- 一个 chunk 控制在 300~600 中文字符
- 不要把整个 SOP 一次塞进一个 chunk
- 将“前置条件 / 步骤 / 风险提示 / 升级条件”分别切开

### 适用于多轮诊断的切分
- 保留命令块和现象描述在同一 chunk
- 保留“症状 → 排查顺序 → 处理建议”链路
- 将“禁止事项”单独建索引，方便做风控判断

## 场景化案例

### 案例 1：RAG 数据导入 在 prod-shanghai-1 的现场处理
- 现象：用户反馈与 `RAG 数据导入` 相关的操作出现异常或体验下降。
- 初始信号：`TopK 召回` 超过基线，值班人在 `chunker` 中看到异常趋势。
- 研判：先排除是否由 `切分过碎` 或最近变更导致。
- 建议动作：先做只读排查，再根据影响范围决定是否回滚、限流、降级或转人工审批。
- 记录要求：记录工单号、环境、服务名、时间窗、关键证据和处理结果。

- 说明：发布后 RAG 数据导入 错误率上升，首先确认最近 2 小时内是否存在版本、配置或依赖变更。
- 说明：当 RAG 数据导入 在 prod-shanghai-1 出现抖动时，应先看 TopK 召回，再决定是否扩容、回滚或转人工。
- 说明：如果 chunker 显示异常，但应用日志没有明显报错，要优先检查监控粒度和依赖项。
- 说明：遇到 切分过碎 时，不要先执行写操作，应先收集证据并同步值班群。
