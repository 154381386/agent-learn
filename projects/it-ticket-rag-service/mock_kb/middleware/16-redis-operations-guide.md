# Redis 使用与排障手册

## 文档属性
- 文档类型：中间件手册
- 适用对象：研发、SRE、缓存平台值班
- 风险等级：中到高
- owner：缓存平台组
- 关键词：`redis`、`cache hit`、`bigkey`、`hotkey`、`eviction`

## 适用范围
适用于公司统一托管的 Redis 集群，包括：
- 业务缓存
- 分布式锁
- 会话存储
- 延迟队列轻量场景

不适用于：
- 自建本地 Redis 容器
- 临时开发环境嵌入式缓存

## 集群类型
| 类型 | 用途 | 特点 |
|---|---|---|
| `redis-standard` | 通用缓存 | 主从 + 哨兵 |
| `redis-cluster` | 大 keyspace 分片 | Cluster 模式 |
| `redis-session` | 会话存储 | 高可用优先 |

## 基本规范
- Key 命名必须带业务前缀
- 需要设置 TTL 的缓存必须显式过期
- 禁止把 Redis 当主数据库使用
- 不允许把超大 JSON 对象整体塞入单 key

## Key 命名建议
```text
order:detail:{order_id}
user:profile:{user_id}
payment:lock:{payment_id}
feature:flag:{flag_name}
```

## 常见指标
- 命中率 `keyspace_hits / (hits + misses)`
- 内存使用率
- 连接数
- 慢查询数量
- `evicted_keys`
- `rejected_connections`

## 常见问题
### 1. 命中率骤降
排查：
- 是否发生大面积 key 过期
- 是否最近发布修改了 key 结构
- 是否切换了序列化格式
- 是否回源 DB 变慢导致缓存击穿

### 2. 内存打满
排查：
- 是否出现 bigkey
- 是否有无 TTL 的长期缓存
- 是否写入模式异常放大
- 淘汰策略是否合理

### 3. 某个接口偶发慢
排查：
- 热 key 是否打到单分片
- 客户端连接池是否不足
- 是否有 pipeline/批量命令错误使用

## 运维排查建议
### 热 key 排查
- 结合客户端埋点与 Redis 监控看单 key QPS
- 必要时评估本地缓存或分片方案

### bigkey 排查
- 观察内存占用异常的时间点
- 对照最近版本是否引入新的缓存字段
- 排查是否把列表/集合无限累积

### 淘汰问题
- 如果 `evicted_keys` 快速增长，说明缓存容量或 TTL 策略需要重新评估
- 必须同时观察 DB/QPS 是否被缓存 miss 放大

## 风险提示
- 禁止在生产上直接执行 `FLUSHALL`
- 禁止未审批修改淘汰策略
- 禁止在线执行不受控的大范围扫描命令

## FAQ
### Q1：缓存 miss 了是不是一定要加大内存？
不一定，先看 key 设计、TTL、热 key 分布和业务读写模式。

### Q2：能不能把排行榜、超大列表直接放 Redis？
可以，但必须评估数据结构、大小、访问模式和过期策略。

### Q3：Redis 延迟高是不是网络问题？
不一定，也可能是热 key、慢命令、连接池不足或淘汰压力。

## 升级条件
- 命中率异常导致 DB 被打穿
- 需要修改 Redis 配置或淘汰策略
- 怀疑缓存集群节点异常

- 说明：如果 MySQL CLI 显示异常，但应用日志没有明显报错，要优先检查监控粒度和依赖项。
