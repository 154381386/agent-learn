# IT Ticket Agent 渐进式重构计划（LangGraph 目标版）

本文档用于指导 `projects/it-ticket-agent` 从当前的“可运行骨架”逐步演进为目标中的 **Supervisor + 多 SubAgent + LangGraph + 中心化 HITL + Executor/Verifier 闭环** 架构。

目标不是一次性推翻重写，而是：

- 保留当前可运行的 HTTP API 与前端控制台
- 用渐进式替换的方式把内部 orchestrator 收敛到 LangGraph
- 在每个阶段都保持“可运行、可验证、可回退”
- 优先固化数据契约和状态模型，再扩展 agent 数量和复杂度

---

## 1. 文档定位

这不是概念草图，而是开发指导文档。它回答以下问题：

- 先改什么，后改什么
- 哪些文件保留，哪些文件重构，哪些文件新增
- 每个阶段的交付物是什么
- 每个阶段完成后如何判断“可以进入下一阶段”
- 哪些地方先兼容，哪些地方后续再彻底收敛

---

## 2. 当前系统现状

### 2.1 当前已有能力

当前项目已经具备以下最小闭环：

- FastAPI HTTP 入口
- Supervisor 路由骨架
- `cicd_agent` / `general_sre_agent` 两个 agent
- MCP Client 与简单连接配置
- RAG HTTP Client
- SQLite 审批存储
- 前端控制台 + 审批弹窗

当前主流程是：

1. 用户请求进入 `/api/v1/tickets`
2. `RuleBasedSupervisor` 选择一个 agent
3. 被选中的 agent 执行并返回 `AgentResult`
4. orchestrator 从 `recommended_actions` 中挑第一个高风险动作
5. 如果有高风险动作则创建审批单
6. 审批通过后 orchestrator 直接调用 MCP 执行动作
7. 返回执行结果给前端

### 2.2 当前主要限制

当前实现距离目标架构仍有明显差距：

- 还是“单 agent 路由”，不是多 subagent 并行
- HITL 虽然中心化，但只有“挑第一个高风险动作”这一层逻辑
- 没有独立的 `approval-coordinator`
- 没有独立的 `executor`
- 没有独立的 `verifier`
- 没有 `IncidentState` 这样的共享状态对象
- agent 输出结构还不是 proposal/facts/hypotheses 风格
- 没有 `k8s-agent`、`observability-agent`、`rca-judge-agent`、`handoff-agent`
- 还没有真正落到 LangGraph `StateGraph + interrupt/resume`

---

## 3. 目标架构定义

### 3.1 最终目标

最终系统应具备以下架构特征：

- `Supervisor` 负责识别意图、规划诊断路径、决定下一步
- 多个 `SubAgent` 负责各领域分析，仅输出结构化结果和 proposal
- `ApprovalCoordinator` 统一收集、去重、合并、排序 proposal，并触发 HITL
- `Executor` 只执行已批准动作，不做自由决策
- `Verifier` 验证动作效果，并把结果写回状态
- `HandoffAgent` 负责转人工与交接包生成
- 全部节点挂在 LangGraph 上，由统一 `IncidentState` 驱动

### 3.2 设计原则

1. **审批中心化**
   - 任何 subagent 都不能自己中断流程发起审批
   - 只有中心审批节点可以 `interrupt`

2. **执行中心化**
   - Subagent 不直接执行高风险动作
   - 执行动作必须经过 executor/gateway

3. **结构化优先**
   - Agent 输出尽量结构化，降低自由文本带来的解析成本

4. **状态驱动**
   - 每个节点只读/写统一状态，不靠隐式上下文传递

5. **渐进兼容**
   - 外部 API 尽量稳定
   - 旧字段先兼容，逐步收敛

## 4. 重构总策略

### 4.1 采用“渐进式重构 + LangGraph 终局”

建议策略：

- **短期**：保留当前 API、前端、MCP/RAG 接口
- **中期**：把内部 orchestrator 改造成 LangGraph 最小闭环
- **长期**：扩展多 agent、审批协调、执行验证、转人工、资源拓扑能力

### 4.2 不建议现在直接重写的原因

如果现在直接新开一个“更正式的实现”，风险包括：

- 需求仍在澄清，容易把不稳定设计过早写死
- 现有可运行闭环会失去参考价值
- 迁移成本会很高，且难以分阶段验证
- 出现问题时很难定位是新架构问题还是迁移问题

因此，推荐做法是：

- 保持现有 HTTP 层与前端不变
- 在 `runtime/` 内部完成架构替换
- 每个阶段都能在当前 UI 上跑通

## 5. 分阶段路线图

整个重构建议拆成 7 个阶段，每个阶段都必须有明确交付物和验收标准。

## 6. Phase 0：冻结现状与补齐契约

### 6.1 目标

在开始 LangGraph 重构之前，先把现状里“容易漂移”的接口收紧，避免后续边迁移边修契约。

### 6.2 交付物

- 统一当前 API 请求/响应契约
- 统一 RAG Client 的兼容行为
- 明确 MCP tool result 的结构要求
- 补齐基本运行文档

### 6.3 必做任务

1. 明确外部 API 稳定面
   - 固定 `POST /api/v1/tickets`
   - 固定 `POST /api/v1/approvals/{approval_id}/decision`
   - 保持前端无需因内部重构而大改

2. 为 RAG 返回结构加 adapter
   - 当前存在 `hits` / `context` 字段不一致问题
   - 在 `rag_client.py` 或专门的 adapter 层统一输出规范字段

3. 统一 ToolExecutionResult 规范
   - 至少保证：`tool_name/status/summary/payload/evidence/risk`

4. 明确 AgentResult 只是“旧世界兼容层”
   - 后续新图里不再把它当最终核心状态
   - 它会逐步退化成 HTTP 响应渲染层使用的 DTO

### 6.4 文件级改动建议

- 保留：`projects/it-ticket-agent/src/it_ticket_agent/main.py`
- 保留：`projects/it-ticket-agent/src/it_ticket_agent/schemas.py`
- 调整：`projects/it-ticket-agent/src/it_ticket_agent/rag_client.py`
- 可新增：`projects/it-ticket-agent/src/it_ticket_agent/adapters/rag_adapter.py`

### 6.5 验收标准

- 无论 RAG 服务返回 `hits` 还是 `context`，上层 tool 都能正常工作
- 当前 UI 仍可完整跑通“发工单 -> 返回诊断 -> 审批 -> 执行结果”

## 7. Phase 1：先建立统一状态模型，而不是先加更多 agent

### 7.1 目标

先把目标架构里的状态对象固定下来。没有统一状态，就无法稳定迁移到 LangGraph。

### 7.2 新增核心概念

新增统一状态模型：`IncidentState`

建议字段如下：

```python
class IncidentState(TypedDict, total=False):
    ticket_id: str
    thread_id: str
    user_id: str
    message: str
    channel: str

    service: str
    cluster: str
    namespace: str
    environment: str

    route: dict
    task_plan: list[dict]
    incident_status: str

    facts: list[dict]
    evidence: list[dict]
    hypotheses: list[dict]

    candidate_actions: list[dict]
    approval_proposals: list[dict]
    approved_actions: list[dict]
    rejected_actions: list[dict]

    execution_results: list[dict]
    verification_results: list[dict]

    subagent_results: list[dict]
    open_questions: list[str]
    handoff_needed: bool
    final_response: dict
    errors: list[dict]
```

### 7.3 配套 DTO

除了总状态，还需要新增结构化 DTO：

- `SubAgentResult`
- `ApprovalProposal`
- `ApprovedAction`
- `ExecutionResult`
- `VerificationPlan`
- `VerificationResult`

### 7.4 为什么这一阶段必须先做

因为当前代码里：

- Supervisor 用的是 `RoutingDecision`
- Agent 用的是 `AgentResult`
- 审批用的是 SQLite record
- 执行结果又是 MCP 原始 payload

这些对象互相之间没有统一边界，后面会让 graph 节点职责混乱。

### 7.5 文件级改动建议

新增建议：

- `projects/it-ticket-agent/src/it_ticket_agent/state/models.py`
- `projects/it-ticket-agent/src/it_ticket_agent/state/incident_state.py`
- `projects/it-ticket-agent/src/it_ticket_agent/state/transformers.py`

保留但逐步弱化：

- `projects/it-ticket-agent/src/it_ticket_agent/runtime/contracts.py`

### 7.6 验收标准

- 新增状态模型能覆盖现有 ticket/approval/agent/tool 数据
- 当前 `cicd_agent` 的输出可以被映射到 `IncidentState`
- 状态转换不依赖前端或 HTTP 层

---

## 8. Phase 2：落地 LangGraph 最小闭环图

### 8.1 目标

把当前自定义 orchestrator 改造成最小可运行 LangGraph 图，但不要一次引入多 agent 并行。

### 8.2 这一阶段的最小图

建议先做这条最短主路径：

`ingest -> supervisor -> single_domain_agent -> approval_gate -> finalize`

如果审批通过，再走：

`approval_gate -> executor -> finalize`

### 8.3 为什么先做最小图

因为你最终目标虽然是多 agent + fan-out，但真正难点不是“并行”，而是：

- 状态如何更新
- approval 如何中断和恢复
- 节点职责如何收敛

先把 LangGraph 的最小串行图跑通，后续再加 fan-out 风险最低。

### 8.4 节点设计建议

#### `ingest_node`
- 把 `TicketRequest` 转成 `IncidentState`
- 初始化默认字段

#### `supervisor_node`
- 只负责路由、任务计划和下一步策略
- 暂时先返回一个主 domain agent

#### `single_domain_agent_node`
- 先复用现有 `CICDAgent` / `GeneralSREAgent`
- 但输出要转换成 `SubAgentResult + Proposal`

#### `approval_gate_node`
- 如果没有 proposal 或没有高风险动作，则直接通过
- 如果有高风险动作，则在这里统一触发 `interrupt`

#### `executor_node`
- 仅执行已经批准的动作

#### `finalize_node`
- 把 graph state 转换成 `TicketResponse`

### 8.5 文件级改动建议

新增：

- `projects/it-ticket-agent/src/it_ticket_agent/graph/builder.py`
- `projects/it-ticket-agent/src/it_ticket_agent/graph/nodes.py`
- `projects/it-ticket-agent/src/it_ticket_agent/graph/edges.py`
- `projects/it-ticket-agent/src/it_ticket_agent/graph/resume.py`

重构：

- `projects/it-ticket-agent/src/it_ticket_agent/runtime/orchestrator.py`

建议保留一个兼容门面：

- `SupervisorOrchestrator` 不立即删除
- 改为内部调用 LangGraph app

### 8.6 验收标准

- 前端和 API 不变
- 内部已经通过 LangGraph 执行主流程
- 单 agent 诊断仍能跑通
- 一次审批动作可以被正常中断和恢复

## 9. Phase 3：把审批升级成真正的 Approval Coordinator

### 9.1 目标

把当前“取第一个高风险动作”的逻辑，升级成真正的 proposal 协调器。

### 9.2 这一阶段要解决的问题

当前问题：

- approval 基于 `recommended_actions`
- 只取第一个高风险动作
- 没有 proposal 池
- 没有 dedupe / merge / policy check

这和目标架构不一致。

### 9.3 新设计

新增 `ApprovalProposal` 作为正式审批输入：

```json
{
  "proposal_id": "p-cicd-1",
  "agent": "cicd-agent",
  "action": "cicd.rollback_release",
  "resource": {
    "service": "order-service",
    "environment": "prod-shanghai-1"
  },
  "params": {
    "target_revision": "release-2026.03.18.2"
  },
  "risk": "high",
  "reason": "发布后失败，需回滚止血",
  "expected_outcome": ["error rate 降低", "服务恢复稳定"],
  "verification_plan": {
    "window_minutes": 5,
    "signals": ["error_rate", "latency_p95"]
  }
}
```

### 9.4 Coordinator 的职责

必须至少实现以下 5 件事：

1. `collect`
   - 收集各 subagent proposal

2. `dedupe`
   - 同 incident + 同 action + 同资源去重

3. `merge`
   - 合并相近 proposal，减少用户多次审批

4. `policy_check`
   - 判定哪些动作必须审批，哪些动作可自动执行

5. `interrupt`
   - 统一触发审批中断

### 9.5 存储升级建议

当前 SQLite 审批表还能继续用，但建议补齐字段：

- `proposal_id`
- `resource_json`
- `expected_outcome_json`
- `verification_plan_json`
- `decision_payload_json`
- `created_at`
- `decided_at`

### 9.6 文件级改动建议

新增：

- `projects/it-ticket-agent/src/it_ticket_agent/approval/models.py`
- `projects/it-ticket-agent/src/it_ticket_agent/approval/coordinator.py`
- `projects/it-ticket-agent/src/it_ticket_agent/approval/policy.py`

重构：

- `projects/it-ticket-agent/src/it_ticket_agent/approval_store.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/orchestrator.py`

### 9.7 验收标准

- agent 返回的是 proposal，而不是直接把“审批语义”塞进 recommended_actions
- approval 节点可以处理多个 proposal
- UI 一次可以展示一个统一审批 payload

## 10. Phase 4：拆出 Executor 与 Verifier，形成闭环

### 10.1 目标

把“审批通过后 orchestrator 直接执行”改成正式闭环：

`approved_actions -> executor -> verifier -> supervisor/finalize`

### 10.2 这一阶段必须新增的能力

#### `executor_node`
- 只执行已批准动作
- 不负责审批、不负责决策、不负责分析
- 通过统一 execution gateway 调 MCP/tool

#### `verifier_node`
- 根据 `verification_plan` 验证动作效果
- 读取 metrics / logs / alerts / rollout / health 信号
- 输出是否恢复、是否需要进一步处理

### 10.3 执行网关建议

建议单独抽象 `ExecutionGateway`：

- 统一封装 MCP 调用
- 屏蔽 tool 名称差异
- 统一治理 metadata，例如：
  - `read_only`
  - `mutating`
  - `requires_approval`
  - `timeout_sec`
  - `retryable`

### 10.4 Verifier 初期实现建议

初期不要追求很重的 observability 验证，可先做轻量版：

- 若回滚后部署状态从 `degraded` 变 `healthy`
- 若活跃告警数量下降
- 若关键错误指标低于阈值

后续再接完整 observability agent。

### 10.5 文件级改动建议

新增：

- `projects/it-ticket-agent/src/it_ticket_agent/execution/gateway.py`
- `projects/it-ticket-agent/src/it_ticket_agent/execution/executor.py`
- `projects/it-ticket-agent/src/it_ticket_agent/verification/verifier.py`
- `projects/it-ticket-agent/src/it_ticket_agent/verification/contracts.py`

### 10.6 验收标准

- 审批通过后，不再由 orchestrator 直接 call MCP
- 执行结果被结构化记录到 state
- 验证结果被结构化记录到 state
- 最终响应能体现“执行了什么、验证结果如何”

## 11. Phase 5：引入多 Agent 并行与共享 IncidentState

### 11.1 目标

从单 agent 迁移到真正的多 subagent 诊断模式。

### 11.2 这一阶段新增的 agent

建议按以下顺序引入，不要同时全上：

1. `observability_agent`
2. `k8s_agent`
3. `rca_judge_agent`
4. `handoff_agent`

原因：

- `observability_agent` 对 verifier 也有复用价值
- `k8s_agent` 是 IT 工单场景中最常见的执行/诊断域
- `rca_judge_agent` 依赖多个 agent 输出，必须后置
- `handoff_agent` 更像流程终点，不必最早引入

### 11.3 并行模式设计

此阶段开始启用 `fan_out`：

`supervisor -> [cicd_agent, k8s_agent, observability_agent] -> aggregate -> approval_coordinator`

### 11.4 聚合节点职责

新增 `aggregate_subagent_results_node`，负责：

- 汇总 `facts/evidence/hypotheses/candidate_actions`
- 合并 `approval_proposals`
- 收敛 `open_questions`
- 标记冲突证据

### 11.5 子 agent 输出格式必须收敛

从这一阶段开始，所有 agent 输出统一为：

- `facts`
- `evidence`
- `hypotheses`
- `candidate_actions`
- `approval_proposals`
- `open_questions`

不再允许只返回“自然语言总结 + recommended_actions”。

### 11.6 文件级改动建议

新增：

- `projects/it-ticket-agent/src/it_ticket_agent/agents/k8s.py`
- `projects/it-ticket-agent/src/it_ticket_agent/agents/observability.py`
- `projects/it-ticket-agent/src/it_ticket_agent/agents/rca_judge.py`
- `projects/it-ticket-agent/src/it_ticket_agent/agents/handoff.py`
- `projects/it-ticket-agent/src/it_ticket_agent/graph/aggregate.py`

重构：

- `projects/it-ticket-agent/src/it_ticket_agent/agents/base.py`
- `projects/it-ticket-agent/src/it_ticket_agent/agents/cicd.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/contracts.py`

### 11.7 验收标准

- 一个工单可以触发多个 agent 并行分析
- 多个 agent 输出能被合并进统一 state
- 多个 proposal 能被 coordinator 收口

## 12. Phase 6：加入 RCA Judge 与 Handoff 闭环

### 12.1 目标

当多个 subagent 的证据冲突时，引入裁决与转人工能力。

### 12.2 `rca_judge_agent` 的职责

- 输入多个 subagent 输出
- 对根因进行排序
- 标注证据冲突点
- 产出优先行动建议

### 12.3 `handoff_agent` 的职责

- 当上下文不足、冲突未解决、动作无效或风险过高时
- 生成人工交接包
- 发送或记录 handoff 事件

### 12.4 人工交接包建议内容

- ticket 基本信息
- 当前影响范围
- 已确认事实
- 已尝试动作
- 执行结果
- 未解决问题
- 建议接手方向

### 12.5 验收标准

- 多 agent 证据冲突时，系统能说明冲突而不是强行拍结论
- 无法自动收敛时能进入 handoff，而不是静默失败

---

## 13. Phase 7：补上实体解析与资源拓扑层

### 13.1 目标

解决自然语言理解和资源定位能力不足的问题，这一层不是 LangGraph 本身，而是 graph 之上的“输入理解基础设施”。

### 13.2 这层要解决什么问题

它要解决以下输入理解问题：

- “支付服务” 实际对应哪个 service id
- “华东 prod” 对应哪个 region/environment
- “订单相关服务” 指的是哪些候选组件
- “订单库挂了影响谁” 需要从数据库节点反查依赖服务

### 13.3 建议拆成两个子系统

#### A. 实体解析层

负责：

- alias normalization
- 模糊匹配
- 资源名补全
- 用户输入结构化

#### B. 资源拓扑层

负责：

- service/db/cache/mq/job 等资源建模
- 正向依赖查询
- 反向依赖查询
- 影响面分析

### 13.4 初期不要做太重

建议先做轻量 MVP：

- 资源注册表
- 别名表
- 模糊搜索接口
- 基础依赖图查询接口

后续再考虑完整 CMDB 或图数据库。

### 13.5 这一阶段与 agent 的关系

未来 supervisor / subagent 不应直接用用户原始文本做所有判断，而应优先消费：

- 已结构化的资源对象
- 已解析的环境信息
- 已召回的候选资源列表
- 拓扑查询结果

### 13.6 文件级改动建议

新增：

## 16. 迁移时的兼容策略

- `POST /api/v1/tickets`
- `POST /api/v1/approvals/{approval_id}/decision`

### 16.2 前端层兼容

前端控制台先不做结构性大改，只逐步增强显示字段：

- diagnosis
- approval payload
- execution summary
- verification summary

### 16.3 Agent 兼容

- 旧：`AgentResult`
- 新：`SubAgentResult / ApprovalProposal`

## 17. 建议的开发顺序（具体到任务批次）

建议按以下批次开发，而不是按“功能名”随意并行：

### 批次 A：契约与状态

1. 引入 `state/` 模型
2. 引入 RAG adapter
3. 为旧 agent 输出增加 transformer
4. 保证现有流程继续可跑

### 批次 B：最小 LangGraph 图

1. 新建 graph builder/nodes
2. 用 graph 跑通 supervisor -> single agent -> finalize
3. 接入 approval gate
4. 保持旧 orchestrator 作为 facade

### 批次 C：审批协调器

1. 定义 proposal 模型
2. 升级 approval store
3. 实现 policy check
4. 接通 interrupt/resume

### 批次 D：执行与验证

1. 加 execution gateway
2. 拆 executor
3. 拆 verifier
4. 响应展示 execution/verification

### 批次 E：多 agent 并行

1. 先加 observability_agent
2. 再加 k8s_agent
3. 做 aggregate 节点
4. 接入 rca_judge_agent

### 批次 F：handoff 与资源解析

1. handoff 流程
2. 实体解析层
3. 资源拓扑层
4. 将 supervisor 与 subagent 输入切换到结构化上下文

---

## 18. 每阶段都要做的验证

每个阶段完成后，至少验证以下场景：

1. **普通咨询场景**
   - 无高风险动作
   - 能返回诊断结果

2. **高风险审批场景**
   - 生成审批单
   - 可批准/拒绝

3. **执行后闭环场景**
   - 批准后能执行
   - 能记录执行结果

4. **异常降级场景**
   - MCP 不可用时系统能降级
   - RAG 不可用时系统能降级
   - LLM 不可用时系统能回退

5. **状态一致性场景**
   - 同一 ticket 在多节点间状态不丢失

## 19. 明确的非目标

为了避免范围失控，以下内容不建议在前几阶段过早做：

- 一开始就接完整真实生产平台
- 一开始就做很重的图数据库方案
- 一开始就做复杂 memory 系统
- 一开始就把所有 agent 同时接入
- 一开始就重写前端控制台

这些都应该在主流程稳定后再做。

## 22. 一句话总结

这次重构的正确顺序不是“先加很多 agent”，而是：

**先统一状态与契约，再落 LangGraph 最小图，再做中心化审批，再做执行验证闭环，最后扩展多 agent 和资源拓扑能力。**

只有按这个顺序推进，项目才会既能持续运行，又能逐步靠近你设想的生产级架构。
