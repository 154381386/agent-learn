# IT Ticket Agent 并行分工方案（适合分配给多个 AI 协作）

本文档用于把 `projects/it-ticket-agent` 的重构工作拆成多个可并行推进的工作流，让不同 AI 可以同时开发、尽量少冲突、方便合并。

这份文档默认基于 `projects/it-ticket-agent/重构计划.md:1` 执行。

---

## 1. 文档目标

这份文档解决 4 个问题：

- 哪些任务可以并行做
- 每个 AI 应该负责哪些目录和文件
- 哪些公共契约要先冻结，避免互相打架
- 合并时如何减少冲突、如何验收

核心原则只有三条：

1. **先冻结契约，再并行开发**
2. **按目录和职责划分所有权**
3. **公共入口少人动，适配层多隔离**

---

## 2. 并行协作总原则

### 2.1 先做“薄公共层”，再做“厚业务层”

最容易冲突的文件通常是：

- `src/it_ticket_agent/main.py`
- `src/it_ticket_agent/runtime/orchestrator.py`
- `src/it_ticket_agent/schemas.py`
- `src/it_ticket_agent/settings.py`
- 各种 `__init__.py`

因此规则是：

- 公共入口文件尽量由 **1 个 AI** 负责
- 其他 AI 尽量在新增目录、新增文件中开发
- 通过 adapter / transformer / facade 接入，不直接多人同时改入口

### 2.2 优先“新增文件”，少量“改旧文件”

为了降低 merge conflict：

- 能新增文件就不要多人改老文件
- 能在 graph/state/approval/execution 这些新目录里做，就不要直接改 `runtime/`
- 老文件改动尽量集中在最后的接线阶段统一完成

### 2.3 每个 AI 只拥有一个主工作流

不要让一个 AI 同时负责：

- 状态模型 + graph builder + approval store + 前端

这样边界太宽，会和所有人冲突。

正确做法是：

- 一个 AI 负责一个垂直工作流
- 只在约定好的接口处对接

---

## 3. 推荐的并行组织方式

建议按 **5 个 AI 工作流 + 1 个集成人** 来分工。

- **AI-1：状态与契约组**
- **AI-2：LangGraph 主流程组**
- **AI-3：审批协调组**
- **AI-4：执行与验证组**
- **AI-5：领域 Agent 与工具组**
- **Integrator：集成与收口组**

如果你只有 3 个 AI，也可以合并：

- AI-A：状态 + Graph
- AI-B：审批 + 执行验证
- AI-C：领域 Agent + 工具 + 资源解析
- 你自己做 Integrator

## 4. 每个角色的职责边界

## 4.1 AI-1：状态与契约组

### 目标

冻结新架构下的核心数据结构，让其他组都围绕同一份契约开发。

### 负责内容

- 定义 `IncidentState`
- 定义 `SubAgentResult`
- 定义 `ApprovalProposal`
- 定义 `ApprovedAction`
- 定义 `ExecutionResult`
- 定义 `VerificationPlan`
- 定义 `VerificationResult`
- 定义旧对象到新状态的 transformer
- 定义 RAG adapter 的标准输出结构

### 主要目录

- `projects/it-ticket-agent/src/it_ticket_agent/state/`
- `projects/it-ticket-agent/src/it_ticket_agent/adapters/`

### 可以修改的旧文件

- `projects/it-ticket-agent/src/it_ticket_agent/rag_client.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/contracts.py`

### 尽量不要碰的文件

- `projects/it-ticket-agent/src/it_ticket_agent/main.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/orchestrator.py`
- `projects/it-ticket-agent/src/it_ticket_agent/static/app.js`

### 交付物

- 可 import 的状态模型
- 兼容旧 `AgentResult` 的 transformer
- 可被其他组依赖的接口说明

### Definition of Done

- 其他组可以只通过状态模型文件完成开发，不需要猜字段名
- `hits/context` 兼容规则明确并已实现
- 至少给出 2-3 个状态示例 payload

---

## 4.2 AI-2：LangGraph 主流程组

### 目标

把现有 orchestrator 替换为 LangGraph 最小闭环，但不直接负责审批存储细节和领域 agent 细节。

### 负责内容

- 建 LangGraph `builder`
- 建 graph nodes / edges
- 建 graph resume 逻辑骨架
- 让旧 `SupervisorOrchestrator` 成为 facade
- 跑通最小链路：
  - `ingest -> supervisor -> domain_agent -> approval_gate -> finalize`

### 主要目录

- `projects/it-ticket-agent/src/it_ticket_agent/graph/`

### 可以修改的旧文件

- `projects/it-ticket-agent/src/it_ticket_agent/runtime/orchestrator.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/supervisor.py`

### 尽量不要碰的文件

- `projects/it-ticket-agent/src/it_ticket_agent/approval_store.py`
- `projects/it-ticket-agent/src/it_ticket_agent/agents/cicd.py`
- `projects/it-ticket-agent/src/it_ticket_agent/static/app.js`

### 依赖输入

- 依赖 AI-1 提供的 `IncidentState` 和 transformer
- 可先用 mock approval/execution 节点，后续再接 AI-3 / AI-4

### 交付物

- LangGraph app builder
- 最小图节点实现
- 从 `TicketRequest` 到 `TicketResponse` 的 graph flow

### Definition of Done

- 当前 HTTP API 不改也能走 graph
- 不依赖前端就能本地调用 graph 完成一次 ticket 流程
- graph 节点职责清晰，不把 approval/execution 细节硬编码进去

## 4.3 AI-3：审批协调组

### 目标

把当前简单审批升级成真正的 `ApprovalCoordinator`，但不负责 graph 主流程和 agent 推理本身。

### 负责内容

- 定义 approval 领域模型
- 设计 proposal collect / dedupe / merge / policy check
- 升级 SQLite approval store
- 定义统一审批 payload
- 负责 resume 后的 decision 结构

### 主要目录

- `projects/it-ticket-agent/src/it_ticket_agent/approval/`

### 可以修改的旧文件

- `projects/it-ticket-agent/src/it_ticket_agent/approval_store.py`
- `projects/it-ticket-agent/src/it_ticket_agent/schemas.py`

### 尽量不要碰的文件

- `projects/it-ticket-agent/src/it_ticket_agent/runtime/supervisor.py`
- `projects/it-ticket-agent/src/it_ticket_agent/agents/`
- `projects/it-ticket-agent/src/it_ticket_agent/static/app.js`

### 依赖输入

- 依赖 AI-1 的 `ApprovalProposal` 契约
- 依赖 AI-2 提供 approval gate 节点接入点

### 交付物

- `ApprovalCoordinator`
- 升级后的审批存储结构
- policy check 模块
- 统一审批请求 DTO / decision DTO

### Definition of Done

- 能处理多个 proposal，而不是只取第一个高风险动作
- 能输出统一审批 payload
- graph/resume 层只需要调用 coordinator，不关心内部细节

## 4.4 AI-4：执行与验证组

### 目标

拆出执行与验证闭环，让批准后的动作不再由 orchestrator 直接调用 MCP。

### 负责内容

- 实现 `ExecutionGateway`
- 实现 `executor`
- 实现 `verifier`
- 定义 execution/verification contracts
- 将 MCP tool 调用包装成结构化执行结果

### 主要目录

- `projects/it-ticket-agent/src/it_ticket_agent/execution/`
- `projects/it-ticket-agent/src/it_ticket_agent/verification/`

### 可以修改的旧文件

- `projects/it-ticket-agent/src/it_ticket_agent/mcp/client.py`
- `projects/it-ticket-agent/src/it_ticket_agent/tools/contracts.py`

### 尽量不要碰的文件

- `projects/it-ticket-agent/src/it_ticket_agent/main.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/supervisor.py`
- `projects/it-ticket-agent/src/it_ticket_agent/static/app.js`

### 依赖输入

- 依赖 AI-1 的 `ApprovedAction / ExecutionResult / VerificationResult`
- 依赖 AI-3 的审批决议结构
- 依赖 AI-2 的 graph 节点接入点

### 交付物

- execution gateway
- executor/verifier 模块
- 可供 graph 使用的节点接口

### Definition of Done

- executor 只执行，不做分析判断
- verifier 只验证，不做审批判断
- 执行与验证结果均可写回 state

## 4.6 Integrator：集成与收口组

### 目标

负责最终接线、冲突合并、冒烟测试和阶段收口。

### 负责内容

- 合并多个 AI 的产物
- 处理跨模块接线
- 控制 `main.py` / `runtime/orchestrator.py` / 前端等高冲突文件的最终修改
- 跑通 end-to-end 场景
- 统一修正 import / 命名 / 契约漂移

### 应由谁担任

优先建议：

- 由你自己担任
- 或者单独指定一个“主 AI”只负责集成，不负责大块功能开发

### Definition of Done

- 每个工作流的产物能拼成一条主流程
- 公共字段名不冲突
- API 与前端能跑通冒烟测试

## 6. 并行开发前必须先冻结的公共契约

在多人/多 AI 并行前，至少先冻结以下内容。

### `IncidentState`

- `ticket_id`
- `thread_id`
- `user_id`
- `message`
- `channel`
- `service`
- `cluster`
- `namespace`
- `route`
- `task_plan`
- `incident_status`
- `facts`
- `evidence`
- `hypotheses`
- `candidate_actions`
- `approval_proposals`
- `approved_actions`
- `rejected_actions`
- `execution_results`
- `verification_results`
- `subagent_results`
- `open_questions`
- `handoff_needed`
- `final_response`
- `errors`

### `ApprovalProposal`

至少冻结：

- `proposal_id`
- `agent`
- `action`
- `resource`
- `params`
- `risk`
- `reason`
- `expected_outcome`
- `verification_plan`

### `ExecutionResult`

至少冻结：

- `action`
- `status`
- `provider`
- `raw_payload`
- `summary`
- `artifacts`
- `error`

### `VerificationResult`

至少冻结：

- `status`
- `summary`
- `signals`
- `window_minutes`
- `passed`

## 6.2 必须先冻结的行为约束

- subagent 不能自己发起审批
- subagent 不能直接执行高风险动作
- approval 只能由 coordinator/approval gate 收口
- executor 不能做自由推理，只能执行已批准动作
- verifier 不能改审批结论，只能产出验证结果

## 7. 推荐的并行开发顺序

虽然目标是并行，但不是所有事情都能同时开始。

建议分成三波。

## 7.1 第一波：可以立即并行

### AI-1

先做：

- `state/` 模型
- `adapters/rag_adapter.py`
- transformer

### AI-2

先做：

- `graph/` 目录
- builder / nodes 骨架
- mock approval / mock executor 节点

### AI-3

先做：

- `approval/` 目录
- proposal model
- coordinator/policy skeleton
- approval store v2 设计草案

### AI-4

先做：

- execution / verification contracts
- execution gateway skeleton
- verifier skeleton

### AI-5

先做：

- `cicd_agent` 输出结构重构方案
- `observability_agent` / `k8s_agent` 文件骨架
- tool 输出规范对齐

这一波可以几乎完全并行。

## 7.2 第二波：开始接线

当 AI-1 的契约稳定后：

- AI-2 开始把 graph 改成用真实 state
- AI-3 开始把 coordinator 接入 graph approval gate
- AI-4 开始把 executor/verifier 接入 graph
- AI-5 开始把 agent 输出对齐到 state/proposal

## 7.3 第三波：集成与收口

由 Integrator 统一完成：

- `runtime/orchestrator.py` facade 接线
- `main.py` 接 graph
- 前端 diagnosis / approval / execution 展示增强
- 端到端验证

```text

【本次改动文件】
- path1
- path2

【新增的公共接口】
```

## 10. 冲突最高的文件与规避方法

## 10.1 高危冲突文件

以下文件冲突概率最高：

- `projects/it-ticket-agent/src/it_ticket_agent/main.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/orchestrator.py`
- `projects/it-ticket-agent/src/it_ticket_agent/schemas.py`
- `projects/it-ticket-agent/src/it_ticket_agent/runtime/supervisor.py`
- `projects/it-ticket-agent/src/it_ticket_agent/static/app.js`
- `projects/it-ticket-agent/src/it_ticket_agent/agents/__init__.py`
- `projects/it-ticket-agent/src/it_ticket_agent/tools/__init__.py`

## 10.2 规避方法

### 方法一：晚改入口

- 各组先在新目录开发
- 最后由 Integrator 一次性接到入口

### 方法二：减少改 `__init__.py`

- 尽量用显式 import 路径
- 不要求每新增模块都立即改 `__init__.py`

### 方法三：不要多人同时碰前端

- 前端 UI 统一最后接线
- 中期阶段只输出后端数据结构，不要求每组都改显示层

### 方法四：不要多人同时改 `schemas.py`

- 新 graph/state 的 DTO 优先放新目录
- `schemas.py` 只保留 HTTP 层模型，由 Integrator 统一收口

## 11. 推荐的分支/提交策略

如果你要把这些工作交给多个 AI，建议采用以下策略：

### 11.1 每个工作流一个分支

例如：

- `refactor/state-contracts`
- `refactor/langgraph-core`
- `refactor/approval-coordinator`
- `refactor/execution-verification`
- `refactor/domain-agents`

### 11.2 不要让每个 AI 从很老的代码分叉太久

建议节奏：

- 每个工作流做一个较小批次就回收
- Integrator 合并后，再让下一轮 AI 基于最新主干继续做

原因：

- AI 分支放太久，最后一定高冲突

### 11.3 每轮并行只做一个批次，不要无限发散

例如第一轮只允许：

- 状态模型
- graph skeleton
- approval skeleton
- execution skeleton
- agent output skeleton

不要第一轮就把 observability/k8s/rca/handoff 全部铺开。

---

## 12. 阶段性验收方式

每一轮并行开发收口时，至少验收以下几项。

## 12.1 契约验收

- 字段名是否与冻结契约一致
- 是否出现两个 AI 各自定义同名不同义对象
- 是否把 DTO 放错层级

## 12.2 接口验收

- graph 是否能调用 approval coordinator
- executor 是否能消费 approved action
- agent 是否能输出 approval proposal

## 12.3 运行验收

至少跑通：

1. 无审批场景
2. 有审批但被拒绝场景
3. 有审批且批准执行场景
4. MCP 不可用降级场景
5. RAG 字段兼容场景

- 不要自己接入 orchestrator
- 不要在 agent 里直接写执行逻辑

### 给 AI-5

任务：

- 让 `cicd_agent` 输出可映射到 proposal/result 的结构
- 为 `observability_agent` 和 `k8s_agent` 建文件骨架
- 统一 tool result 的结构

不要做：

- 不要自己实现审批 coordinator
- 不要自己改 `main.py`

---

## 14. 什么时候不适合并行

以下情况不要并行推进，否则只会互相覆盖：

- `IncidentState` 字段名还没定
- `ApprovalProposal` 契约还没定
- LangGraph 主图入口还没定
- 你还没决定哪些文件由 Integrator 独占

如果这些没冻结，先做半天契约收敛，比直接让 5 个 AI 一起写代码更省时间。

---

## 15. 一句话总结

最稳的分工方式不是“按功能随便分”，而是：

**一个 AI 冻结状态与契约，一个 AI 搭 LangGraph 主流程，一个 AI 做审批协调，一个 AI 做执行验证，一个 AI 做领域 agent，最后由一个集成人统一接线。**

只要坚持“目录所有权 + 公共契约先冻结 + 高冲突文件由集成人独占”这三条，不同 AI 就能并行干活，而且冲突会小很多。
