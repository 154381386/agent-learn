## [LD][2026-03-28] 当前代码审查结论

### 已确认的结构问题
- `src/it_ticket_agent/main.py` 目前先调用 `approval_store.decide()`，再进入 `handle_approval_decision()`；一旦后续执行失败，审批状态会先落库，流程状态与真实执行结果可能不一致。
- `src/it_ticket_agent/approval_store.py` 当前 `decide()` 没有“仅允许 pending -> approved/rejected”的状态保护，也没有审计轨迹，重复审批会直接覆盖旧结果。
- `src/it_ticket_agent/rag_client.py` 仍是远端返回透传；`src/it_ticket_agent/tools/cicd.py` 却主要读取 `context` 或 `hits` 的局部约定，`hits/context` 契约未统一，知识检索结果存在被静默丢弃的风险。
- `src/it_ticket_agent/tools/cicd.py` 的 `SearchKnowledgeBaseTool` 捕获异常后仍返回成功态空结果，会把知识库故障伪装成“未命中”，不利于上层判断。
- `src/it_ticket_agent/runtime/orchestrator.py` 仍只取第一个高风险动作 `_first_risky_action()`，与后续 `proposal collect / dedupe / merge` 目标架构不一致。
- `src/it_ticket_agent/runtime/orchestrator.py` 在审批通过后直接 new `MCPClient` 执行动作，说明执行仍耦合在 orchestrator，后续必须退化为 approval gate / executor 接线点。
- `src/it_ticket_agent/agents/cicd.py` fallback 分支即使生成高风险回滚建议，`risk_level` 仍固定为 `low`，状态表达不一致。

### 对三组的直接要求
- AI-1：优先解决 `hits/context` 统一输出问题，并冻结新状态契约，禁止直接改 HTTP 层。
- AI-2：先做 graph 骨架和 facade，不得继续扩散 orchestrator 的审批/执行逻辑。
- AI-3：审批存储升级时必须补上状态机约束、审计字段和多 proposal 结构，不能沿用“单动作单记录”的思路直接放大。
