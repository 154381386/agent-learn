# Pod 异常排查 Runbook

## 文档属性
- 文档类型：故障排查 Runbook
- 适用对象：研发、SRE、值班同学
- 风险等级：中到高
- owner：SRE 值班体系
- 关键词：`OOMKilled`、`CrashLoopBackOff`、`ContainerCreating`

## 适用症状
- `CrashLoopBackOff`
- `OOMKilled`
- `Error`
- `ContainerCreating` 长时间不结束
- `ImagePullBackOff`
- Pod 频繁重启
- 探针失败导致服务不 Ready

## 排查原则
- 先判断是否影响业务，再决定是否需要止血
- 先收集证据，再执行写操作
- 先看单 Pod 现象，再判断是否集群级问题
- 先确认是否与最近变更有关

## 标准排查顺序
1. 看状态：`kubectl get pod`
2. 看事件：`kubectl describe pod`
3. 看日志：`kubectl logs --previous`
4. 看资源：`kubectl top pod`
5. 看探针：Deployment / PodSpec
6. 对照最近发布和配置变更
7. 判断是否需要回滚、扩容或拉值班

## 常用命令
```bash
kubectl -n default get pod order-service-xxxxx -o wide
kubectl -n default describe pod order-service-xxxxx
kubectl -n default logs order-service-xxxxx --tail=200
kubectl -n default logs order-service-xxxxx --previous --tail=200
kubectl -n default top pod order-service-xxxxx
kubectl -n default get events --sort-by=.lastTimestamp | tail -n 30
```

## 典型场景一：OOMKilled
### 判断依据
- `Last State: Terminated`
- `Reason: OOMKilled`
- 内存曲线长期贴近或撞上 `limits`
- 重启往往发生在流量高峰或发版后

### 重点排查
- 最近是否发版
- 连接池、缓存、本地队列是否放大
- JVM/Runtime 参数是否调整
- 是否出现异常对象堆积

### 处理建议
- 短期：回滚最近版本或临时提高 limit
- 中期：定位内存泄漏、限制批量大小、优化缓存策略
- 长期：做 heap dump 分析和压测回归

## 典型场景二：CrashLoopBackOff
### 重点看
- 启动日志
- 环境变量 / Secret / ConfigMap 是否缺失
- 下游依赖是否不可用
- 初始化脚本是否失败
- 探针是否过于激进

### 常见原因
- 新配置没有下发到目标环境
- 应用启动期依赖数据库，但数据库白名单没开
- 启动耗时变长，readiness/liveness 配置不合理
- 新镜像缺文件或 entrypoint 错误

## 典型场景三：ContainerCreating
### 重点看
- 镜像拉取失败
- Secret / ConfigMap 挂载失败
- PVC 绑定失败
- 节点资源不足
- CNI / 网络初始化异常

### 排查命令
```bash
kubectl -n default describe pod order-service-xxxxx
kubectl get pvc -n default
kubectl get node
```

## 典型场景四：ImagePullBackOff
可能原因：
- 镜像 tag 不存在
- 仓库凭据失效
- 私有镜像仓库不可达
- 发布平台推送失败

处理方式：
- 先在发布平台确认镜像是否存在
- 再确认拉取密钥和 registry health

## 探针相关问题
### readiness 失败表现
- Pod Running 但不 Ready
- Service 没有 Endpoints
- 上游请求报连接失败或 503

### liveness 失败表现
- Pod 反复被 kubelet 杀死重启
- 业务明明启动慢，但健康检查门槛太高

### 调整原则
- 启动较慢的服务要增加 `initialDelaySeconds`
- read 与 live 的超时不要一刀切
- 不要把依赖性很强的深度探测直接放到 liveness

## 是否需要立刻回滚？
满足以下情形，优先考虑回滚：
- 问题与最近发版高度相关
- 故障范围持续扩大
- 短时间内难以通过配置修复
- 当前值班人无法快速确定根因

## 何时不建议立刻回滚
- 不是版本问题，而是下游依赖故障
- 新旧版本共用相同有问题配置
- 需要先确认数据兼容性

## FAQ
### Q1：Pod 重启是不是一定是应用问题？
不是。也可能是节点资源紧张、探针误杀、镜像拉取异常或挂载失败。

### Q2：只看当前日志够不够？
不够。很多情况下必须看 `--previous` 才能看到上一次容器退出前的关键日志。

### Q3：服务挂了先删 Pod 能不能解决？
可能暂时恢复，但这通常不是根因修复。生产环境不能把删 Pod 当标准操作。

## 升级条件
- 同一服务 10 分钟内超过 5 个 Pod 异常
- 影响核心业务链路
- 需要修改 Deployment / 回滚 / 重启
- 怀疑平台层（镜像仓库、节点、CNI）异常

## 误判案例
### 把探针失败误判成代码崩溃
实际上应用没有退出，只是 readiness 连续失败，导致上游看起来像服务挂了。

### 把 OOM 误判成节点问题
如果多个节点、同服务新版本都出现类似内存曲线，优先怀疑应用版本或配置，而不是节点本身。
