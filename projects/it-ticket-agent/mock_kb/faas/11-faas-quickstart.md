# FaaS 平台使用手册

## 文档属性
- 文档类型：平台快速上手
- 适用对象：研发、平台工程师、SRE
- owner：FaaS 平台组
- 关键词：`function`、`cron`、`webhook`、`faasctl`

## 平台入口
- 控制台：`https://faas.internal.example.com`
- CLI：`faasctl`
- 统一日志：接入日志平台 `app-prod-*`

## 适用场景
- 定时任务
- 轻量 webhook 处理
- 文件转换
- 异步通知
- 临时数据修复脚本（需审批）

## 不适用场景
- 长连接服务
- 高状态依赖服务
- 长时间运行的大作业
- 强依赖本地磁盘的任务

## 创建函数流程
1. 选择 Runtime：`python3.11` / `nodejs20`
2. 填写函数名，如 `order-cleanup-job`
3. 上传代码包或绑定 Git 仓库
4. 配置超时、内存、环境变量
5. 绑定触发器：HTTP / Cron / MQ
6. 发布到 `staging` 验证
7. 通过后发生产

## CLI 示例
```bash
faasctl login
faasctl function deploy order-cleanup-job --runtime python3.11 --memory 512 --timeout 60
faasctl function invoke order-cleanup-job --payload '{"date":"2026-03-14"}'
```

## 资源建议
- 默认内存：`256Mi`
- 默认超时：`30s`
- CPU 密集任务建议独立评估
- 高 I/O 任务建议拆成异步工作流

## 观察项
- 调用成功率
- P95 执行时长
- 并发数
- 失败重试次数
- 冷启动次数

## 风险提示
- 不适合长驻连接型任务
- 涉及生产数据修复的函数必须审批
- 不要在环境变量中明文存生产数据库密码

## FAQ
### Q1：FaaS 适合做批量修复吗？
适合小规模、幂等、受控任务；大规模批处理建议交给专门作业平台。

### Q2：函数超时怎么办？
优先拆小任务或改成异步模式，不建议简单拉大超时。

### Q3：如何排查函数偶发失败？
先看重试日志、依赖超时、冷启动和资源限制，再看代码异常。

## 场景化案例

### 案例 1：payment-retry-job 在 prod-shanghai-1 的现场处理
- 现象：用户反馈与 `payment-retry-job` 相关的操作出现异常或体验下降。
- 初始信号：`冷启动次数` 超过基线，值班人在 `函数控制台` 中看到异常趋势。
- 研判：先排除是否由 `重复执行` 或最近变更导致。
- 建议动作：先做只读排查，再根据影响范围决定是否回滚、限流、降级或转人工审批。
- 记录要求：记录工单号、环境、服务名、时间窗、关键证据和处理结果。

- 说明：发布后 payment-retry-job 错误率上升，首先确认最近 2 小时内是否存在版本、配置或依赖变更。
- 说明：当 payment-retry-job 在 prod-shanghai-1 出现抖动时，应先看 冷启动次数，再决定是否扩容、回滚或转人工。
- 说明：如果 函数控制台 显示异常，但应用日志没有明显报错，要优先检查监控粒度和依赖项。
- 说明：遇到 重复执行 时，不要先执行写操作，应先收集证据并同步值班群。
