# Kafka 使用与排障手册

## 文档属性
- 文档类型：中间件手册
- 适用对象：研发、SRE、消息平台值班
- owner：消息平台组
- 关键词：`kafka`、`lag`、`consumer group`、`rebalance`

## 使用场景
- 订单异步通知
- 事件驱动链路
- 审计流水
- 营销与埋点数据传输

## 基本规范
- Topic 命名必须带业务域前缀
- 核心 Topic 必须配置 owner 和消费组列表
- 消费逻辑必须幂等
- 不能依赖“至少一次投递”去保证业务唯一性

## 关键指标
- Produce TPS
- Consume TPS
- Consumer Lag
- Rebalance 次数
- Broker 磁盘使用率
- ISR 缩减次数

## 常见问题
### Lag 持续升高
排查：
- 消费者是否变慢
- 是否发生 rebalance
- 下游依赖是否阻塞消费线程
- 是否有热点 partition

### 重复消费
排查：
- 消费逻辑是否幂等
- offset 提交时机是否正确
- 是否在失败重试中重复执行副作用

### Producer 超时
排查：
- Broker 健康度
- 网络延迟
- acks 配置
- 批量大小和缓冲区设置

## FAQ
### Q1：lag 高是不是一定要扩容消费者？
不一定，也可能是单条消息处理太慢或下游阻塞。

### Q2：消息丢失怎么排查？
先区分“没发送成功”“已发送未消费”“消费了但业务没落地”。

### Q3：能不能手工改 offset？
可以，但必须非常谨慎，通常只在事故场景由平台/SRE执行。

## 升级条件
- 需要重置消费组 offset
- 怀疑 broker 集群异常
- 核心 Topic 出现大面积积压

## 场景化案例

### 案例 1：mysql-primary 在 prod-shanghai-1 的现场处理
- 现象：用户反馈与 `mysql-primary` 相关的操作出现异常或体验下降。
- 初始信号：`命中率` 超过基线，值班人在 `MySQL CLI` 中看到异常趋势。
- 研判：先排除是否由 `慢 SQL` 或最近变更导致。
- 建议动作：先做只读排查，再根据影响范围决定是否回滚、限流、降级或转人工审批。
- 记录要求：记录工单号、环境、服务名、时间窗、关键证据和处理结果。

- 说明：发布后 mysql-primary 错误率上升，首先确认最近 2 小时内是否存在版本、配置或依赖变更。
- 说明：当 mysql-primary 在 prod-shanghai-1 出现抖动时，应先看 命中率，再决定是否扩容、回滚或转人工。
- 说明：如果 MySQL CLI 显示异常，但应用日志没有明显报错，要优先检查监控粒度和依赖项。
- 说明：遇到 慢 SQL 时，不要先执行写操作，应先收集证据并同步值班群。
