# 密钥与配置管理手册

## 文档属性
- 文档类型：安全规范
- 适用对象：研发、SRE、平台工程师
- 风险等级：高
- owner：安全平台组
- 关键词：`secret`、`config`、`rotation`、`env`

## 配置分类
- 普通配置：配置中心管理
- 敏感配置：Secret Manager 管理
- 强敏感配置：审批后下发
- 本地开发配置：仅限开发环境，不可复制到生产

## 基本原则
- 配置与密钥分离
- 默认最小暴露范围
- 所有生产密钥必须可轮换
- 所有敏感配置必须有 owner 和用途说明

## 禁止事项
- 禁止把生产密钥写进代码仓库
- 禁止在群聊里发送完整 AccessKey / Token
- 禁止把 `.env` 文件上传到公共网盘
- 禁止把生产密码写死在 CI 脚本或 Helm values

## Secret 使用规范
- 一个服务一组 Secret，不跨服务混用
- Secret 需要标注 owner、用途、过期时间
- 生产 Secret 轮换后要验证灰度环境再放全量
- 敏感字段在日志中必须脱敏

## 配置发布流程
1. 提交配置变更申请
2. 在 `staging` 验证
3. 经审批后发生产
4. 观察配置加载结果与应用行为
5. 记录配置版本号

## 故障排查
### 启动失败
排查：
- Secret 是否存在
- 挂载路径是否正确
- 环境变量名是否匹配
- Pod 是否重新加载配置

### 新配置未生效
排查：
- 配置是否发布到正确环境
- 是否触发重载或重启
- 是否有本地缓存

## 轮换建议
- 数据库密码：90 天
- 第三方接口密钥：90~180 天
- 临时 Token：小时级或天级

## FAQ
### Q1：为什么不建议把配置写在镜像里？
因为镜像不可区分环境，敏感配置应与运行环境解耦。

### Q2：生产 `.env` 能不能直接发给同事排查？
不能。应通过受控方式按需查看，不允许直接扩散。

### Q3：Secret 修改后为什么应用还在用旧值？
需要确认挂载方式、应用加载方式以及是否需要重启/热更新。

## 场景化案例

### 案例 1：数据库只读账号 在 prod-shanghai-1 的现场处理
- 现象：用户反馈与 `数据库只读账号` 相关的操作出现异常或体验下降。
- 初始信号：`高危命令拦截数` 超过基线，值班人在 `MFA` 中看到异常趋势。
- 研判：先排除是否由 `越权访问` 或最近变更导致。
- 建议动作：先做只读排查，再根据影响范围决定是否回滚、限流、降级或转人工审批。
- 记录要求：记录工单号、环境、服务名、时间窗、关键证据和处理结果。

- 说明：发布后 数据库只读账号 错误率上升，首先确认最近 2 小时内是否存在版本、配置或依赖变更。
- 说明：当 数据库只读账号 在 prod-shanghai-1 出现抖动时，应先看 高危命令拦截数，再决定是否扩容、回滚或转人工。
- 说明：如果 MFA 显示异常，但应用日志没有明显报错，要优先检查监控粒度和依赖项。
- 说明：遇到 越权访问 时，不要先执行写操作，应先收集证据并同步值班群。
