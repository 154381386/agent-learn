把当前 runtime 的会话、审批、执行、事件、记忆等状态存储，从“直接绑定 SQLite 实现”逐步迁移为“可切换 SQLite / Postgres 的存储后端”。

## 配置

```bash
STORAGE_BACKEND=sqlite
APPROVAL_DB_PATH=./data/approvals.db
```

切换到当前已支持的 Postgres 后端：

```bash
STORAGE_BACKEND=postgres
POSTGRES_DSN=postgresql://user:password@host:5432/it_ticket_agent
APPROVAL_DB_PATH=./data/approvals.db
```

说明：

- 当前 `postgres` 模式下，`session`、`system_event`、`approval`、`interrupt`、`checkpoint`、`execution`、`process_memory`、`incident_case` 会切换到 PG
- 其他 store 仍继续使用 `APPROVAL_DB_PATH` 指向的 SQLite
- 因此当前不是最终态，而是过渡态

## 分阶段迁移顺序

### Phase 1：Backend 抽象

- 新增 provider 与 backend 选择
- 保持 SQLite 为默认
- 先迁移读写频率高、结构清晰、依赖面小的表

状态：已完成

### Phase 2：Session / Event 迁移

- `conversation_session`
- `conversation_turn`
- `system_event`

状态：已完成第一版 PG 实现

### Phase 3：Interrupt / Approval 迁移

- `interrupt_request`
- `approval_request_v2`
- `approval_audit_event`

原因：

- 它们是会话恢复与人工交互的关键路径
- 事务语义要求高，应单独迁移和验证

状态：已完成第一版 PG 实现

### Phase 4：Execution / Checkpoint 迁移

- `execution_plan`
- `execution_step`
- `execution_checkpoint`

原因：

- 执行轨迹依赖较多
- 需要重点验证事务一致性和恢复逻辑

状态：已完成第一版 PG 实现

### Phase 5：Memory / IncidentCase / RankerWeights 迁移

- `process_memory_entry`
- `incident_case`
- `ranker_weight_snapshot`

原因：

- 这批更偏分析与沉淀
- 迁移风险相对小，但要和反馈闭环、自适应权重一起验证

状态：

- `process_memory` / `incident_case`：已完成第一版 PG 实现
- `ranker_weight_snapshot`：已支持 PG backend

## 表设计建议

### SQLite 到 PG 的统一原则

- `*_json text` -> `jsonb`
- 时间字段 -> `timestamptz`
- 保持主键为 `text`
- 优先使用 `text + check`，暂不急着改 enum
- 所有 upsert 改为 `insert ... on conflict`

## 风险点

### 1. 混合存储阶段的数据分裂

当 `session/system_event` 已在 PG，而其他表仍在 SQLite 时：

- 调试时需要同时查看两套存储
- 备份与恢复流程会更复杂

### 2. 事务边界不一致

SQLite 当前很多写操作是单表短事务。

迁到 PG 后，需要重新检查：

- 审批状态更新
- 中断恢复
- 执行步骤写入
- checkpoint 回填

### 3. JSON 语义变化

SQLite 当前依赖“写入前手动 `json.dumps`”。

PG 迁移后：

- 需要统一 `jsonb` 写入方式
- 要避免 Python dict / json string 混用

## 完成标准

- 所有 runtime store 都不再直接依赖 `sqlite3`
- 可通过配置切换 `sqlite` / `postgres`
- 核心链路在 PG 模式下通过现有回归测试
- 提供最小数据迁移脚本，支持从 SQLite 导入 PG

## 迁移脚本

  --sqlite-path ./data/approvals.db \
  --postgres-dsn postgresql://user:password@host:5432/it_ticket_agent
