# IT Ticket Agent 当前架构

## 文档定位

本文档描述 `projects/it-ticket-agent` 当前已经落地、默认启用的真实架构。

- 代码主线：`react_tool_first`
- 适用范围：当前仓库 `it-ticket-agent`
- 不包含：历史 `hypothesis_graph + skill` 方案、设计中的 subagent 方案

如果其他文档与本文冲突，以本文和当前代码实现为准。

---

## 1. 当前系统边界

当前仓库负责：

- 会话接入与状态持久化
- Smart Router 分流
- Tool-First ReAct 诊断主循环
- HITL 审批、恢复、执行
- 案例回写、反馈闭环
- 离线评估与 mock / world-state 仿真

当前仓库不再负责：

- 把 `skill` 作为默认诊断编排核心
- 把 `hypothesis_graph` 作为默认主图
- 把多 subagent 作为当前运行时默认能力

RAG 已拆到兄弟项目 `projects/it-ticket-rag-service`，当前仓库通过 `RAGServiceClient` 调用。

---

## 2. 默认主链

`SupervisorOrchestrator` 初始化时会强制把运行模式收敛到 `react_tool_first`。

当前默认主链可以概括为：

```text
HTTP / Conversation API
  -> request normalize
  -> slot resolution / clarification gate
  -> retrieve RAG context
  -> React Graph
       light_router
         -> direct_answer
         -> supervisor_loop
              -> approval_gate
                   -> await_user
                   -> execute_approved_action
              -> finalize
  -> session / checkpoint / case persistence
  -> optional feedback resume
```

需要特别说明两点：

- `context_collector` 是 `supervisor_loop` 内部第一步，不是当前 React Graph 的独立节点。
- `feedback` 当前也不是 graph 节点，而是 `finalize` 阶段创建 interrupt，后续由 orchestrator 在 resume 时处理。

## 3. 端到端流程

### 3.1 请求进入与前置澄清

入口在 `runtime/orchestrator.py` 的 `_run_ticket_message(...)`。

在进入 graph 之前，系统会先做：

1. 规范化 `TicketRequest`
2. `resolve_slots(...)` 做实体识别与槽位补全
3. 如果缺少继续诊断所需信息，则直接创建 `clarification interrupt`
4. 如果信息足够，则拉取 RAG 上下文并构造 graph 输入

也就是说，当前“是否先澄清”发生在 graph 外层，而不是 supervisor 内部临时补问。

### 3.2 Light Router

React Graph 的起点是 `light_router`：

- `direct_answer`：FAQ / SOP / 知识咨询
- `supervisor_loop`：需要排查、分析或执行动作的问题

这里的路由实现依赖 `SmartRouter` 和 `QueryClassifier`，属于轻路由，不承担复杂推理。

### 3.3 Direct Answer

`direct_answer` 复用 `SmartRouter.generate_direct_answer(...)`：

- 有 `rag_context.direct_answer` 时直接返回
- 有 LLM 且 RAG 命中时，基于知识片段生成简洁回答
- 没有足够知识时走 fallback

这条路径不进入工具诊断、不进入审批链。

### 3.4 Supervisor Loop

`supervisor_loop` 是当前系统的主脑，入口实现位于 `runtime/react_supervisor.py`。

内部流程如下：

1. 先调用 `legacy_nodes.context_collector(...)` 补齐上下文
2. 根据 `request + matched_tool_domains + observations` 计算候选问题域
3. 根据候选问题域裁剪本轮可见工具集
4. 进入 LLM ReAct 循环，或在无 LLM 时走 rule-based fallback
5. 将工具结果写入 `observation_ledger`
6. 持续维护：
   - `working_memory_summary`
   - `pinned_findings`
   - `tool_cache`
   - `expanded_domains_seen`
7. 在证据足够、预算耗尽或达到护栏时停止

当前 supervisor 不是“全量工具开放 + 任意探索”，而是明显做了候选收敛：

- 先根据显式信号确定主域，如 `k8s / cicd / network / db / sde`
- 首轮优先暴露 domain-native tools
- 证据不足时才扩到相邻域
- 扩域时优先执行少量 expansion probe tools
- 如果 LLM 请求了本轮未开放的工具，会被记为 `rejected_tool_call`

### 3.5 Tool Execution Middleware

所有工具执行都收口到 `execution/tool_middleware.py`。

当前职责包括：

- 校验 tool 是否注册
- 根据 `risk_level` 判断是否需要审批
- 执行 `timeout / retry`
- 统一错误与执行结果信封

当前规则非常明确：

- `low / medium` 风险：可直接执行
- `high / critical` 风险：返回 `approval_required`，进入审批链

### 3.6 Approval / Execute / Resume

当前 React Graph 中与审批相关的节点是：

- `approval_gate`
- `await_user`
- `execute_approved_action`

语义如下：

- `approval_gate`：把高风险动作转成审批请求
- `await_user`：给响应打上 `resume_target`，真正的等待由 orchestrator 的 session + interrupt 层处理
- `execute_approved_action`：审批通过后的实际执行

恢复路径由 orchestrator 控制：

- `clarification` resume -> 回到 `supervisor_loop`
- `approval` resume -> 进入 `execute_approved_action`
- `feedback` resume -> 不再重跑 graph，直接写回案例库

### 3.7 Finalize 与反馈闭环

`finalize` 负责收敛最终响应。

如果本轮已经形成最终结果，系统会：

- 回写 session 状态
- 写 checkpoint
- 写 process memory / system events
- upsert `IncidentCase`
- 在合适场景下创建 `feedback interrupt`

反馈不是当前主图里的一个独立节点，而是 finalize 后的异步人工确认闭环：

- 用户确认本次根因是否正确
- 写入 `human_verified`
- 写入 `hypothesis_accuracy`
- 写入 `actual_root_cause_hypothesis`
- 如启用向量索引，则重新索引案例

## 4. 当前 React Graph

当前图定义在 `graph/react_builder.py`，真实节点只有 7 个：

```text
START
  -> light_router
      -> direct_answer
      -> supervisor_loop
           -> finalize
           -> approval_gate
                -> await_user
                -> execute_approved_action
                -> finalize
  -> END
```

节点职责：

| 节点 | 当前职责 |
|------|----------|
| `light_router` | 根据 `SmartRouterDecision` 选择 fast path 或诊断主链 |
| `direct_answer` | FAQ / SOP / 知识问答直答 |
| `supervisor_loop` | Tool-first ReAct 推理、工具调用、证据收敛 |
| `approval_gate` | 审批拦截与审批请求构造 |
| `await_user` | 将等待状态回传给 orchestrator，并标记 resume target |
| `execute_approved_action` | 审批通过后的执行 |
| `finalize` | 最终响应收敛与 feedback interrupt 创建 |

## 5. 核心状态对象

### 5.1 TicketRequest

请求对象定义在 `schemas.py`，当前已经支持评估与仿真所需字段：

- `mock_scenario`
- `mock_scenarios`
- `mock_tool_responses`
- `mock_world_state`

### 5.2 IncidentState

`IncidentState` 是当前运行时状态核心，承载：

- 用户原始问题与解析出的实体
- routing 结果
- `rag_context`
- `context_snapshot`
- 审批、执行、反馈相关元数据
- `react_runtime` 与证据摘要

### 5.3 ReactTicketGraphState

当前 React Graph 的运行态字段主要包括：

- `route_decision`
- `response`
- `approval_request`
- `resume_target`
- `observation_ledger`
- `tool_calls_used`
- `working_memory_summary`
- `pinned_findings`
- `stop_reason`

## 6. Context Collector 的真实角色

当前 `context_collector` 仍然存在，但它的角色已经收缩为 supervisor 的内部上下文准备步骤，而不是旧架构中的独立固定流水线入口。

当前输出重点是：

- `similar_cases`
- `matched_tool_domains`
- `retrieval_expansion`
- `context_quality`

当前 `available_tools` 在 `ContextSnapshot` 中仍保留字段，但默认并不承载旧的 `skill` 规划语义；工具暴露主要由 `ReactSupervisor` 的候选集收敛逻辑决定。

## 7. 评估与仿真架构

当前评估目标不是“mock 掉整个 agent”，而是：

- 保持真实 LLM 开启
- 只在工具边界注入可控世界
- 评估最终结论和搜索过程

当前有三条评估线：

### 7.1 静态 Tool Mock 数据集

- 数据集：`data/evals/tool_mock_cases.json`
- 作用：稳定回归、精确控制单个 tool 返回
- 当前规模：13 个 case

### 7.2 Shared World 数据集

- 数据集：`data/evals/world_cases.json`
- 投影器：`testing/world_simulator.py`
- 作用：用同一份 `world_state` 投影多个域工具结果，更接近真实事故现场
- 当前规模：5 个 case

### 7.3 Session Flow 数据集

- 数据集：`data/evals/session_flow_cases.json`
- 作用：覆盖 `clarification / approval / feedback / topic shift` 这类多轮中断恢复链路
- 当前规模：7 个 case
- 运行方式：仍走 `scripts/run_agent_eval.py`，由脚本自动识别为 `session_flow` 模式
- 当前口径：默认按 case 配置 `llm_mode=disabled`，优先回归 orchestrator / interrupt / session 状态机

### 7.4 Mock 优先级

当前工具 mock 优先级为：

1. `mock_response` / `mock_tool_responses`
2. `mock_world_state`
3. `mock_case`
4. `mock_scenario` / profile

### 7.5 当前过程指标

`agent_eval` 除了看 pass / fail，还会记录：

- `tool_calls_used`
- `stop_reason`
- `expanded_domains`
- `expansion_probe_count`
- `rejected_tool_call_count`

`session_flow_eval` 额外看：

- `response_status`
- `session_status`
- `session_stage`
- `pending_interrupt_type`
- `new_system_event_types`
- `new_approval_event_types`
- `case_exists / human_verified / actual_root_cause_hypothesis`

这套指标对应的是当前 supervisor 的真实优化方向：候选收敛、扩域控制、提前停止，而不是旧的 hypothesis 排名质量。

## 8. 持久化与观测

当前 orchestrator 会维护这些运行资产：

- `SessionStore`
- `CheckpointStore`
- `ProcessMemoryStore`
- `ExecutionStore`
- `ApprovalStore`
- `InterruptStore`
- `SystemEventStore`
- `IncidentCaseStore`

当前观测能力主要体现在：

- 路由、上下文采集、direct answer、审批等关键阶段会写 system event
- process memory 记录本轮摘要与关键转移
- observability trace context 会附着到响应

当前项目已经具备基础可追踪性，但系统级 OTel 统一观测并不是当前主线重点。

---

## 9. 哪些东西不是当前架构

以下内容容易在旧文档里出现，但都不应再被理解为“当前默认实现”：

- `hypothesis_graph`
- `Hypothesis Generator`
- `VerificationAgent`
- `skill-first fixed pipeline`
- `feedback_gate` 作为 graph 节点
- `subagent` 已上线

当前仓库里仍可能保留部分相关目录、模型或历史文档，用于迁移记录、兼容字段或未来设计；但默认运行时主链已经不依赖这些概念。
