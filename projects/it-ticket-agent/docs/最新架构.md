
# Robert 架构演进：假设驱动的根因分析系统

> **Multi-Hypothesis Parallel Verification Architecture**
>
> 将现有 Supervisor 领域路由 + 多 Agent 分工模式，演进为 Hypothesis Generator + VerificationAgent 竞争验证模式，统一为两条路径的简洁架构。

---

## 1. 背景与动机

### 1.1 现有架构的局限

当前 Robert 采用 **Supervisor 领域路由 + 多 Domain Agent 分工** 模式：

- `RuleBasedSupervisor` 通过关键词/LLM 路由到 `CICDAgent`、`DBAgent`、`NetworkAgent` 等领域 Agent
- 每个领域 Agent 独立推理 + 执行工具调用
- `Aggregator` 合并多 Agent 结果

存在的问题：

| 问题 | 说明 |
|------|------|
| 路由瓶颈 | 模糊问题（"服务偶尔 502"）难以准确路由到某个领域 |
| 领域视角局限 | 每个 Agent 只从自己领域分析，无法全局推理跨领域根因 |
| 上下文重复传输 | N 个 Agent 各自接收完整上下文，Token 浪费 |
| 结果冲突难调和 | Aggregator 只能合并，无法判断谁的结论更可信 |
| 缺乏反馈闭环 | 诊断结果无法回写案例库，系统不会从历史中学习 |

### 1.2 目标架构

借鉴 **多假设并行验证（Multi-Hypothesis Parallel Verification）** 模式：

- 1 个 Generator 全局推理，生成多个根因假设
- 统一的 VerificationAgent 按验证计划并行执行
- Ranker 按置信度排序，用证据说话
- 人工反馈闭环，持续优化假设质量

核心转变：**智能从分散的 Agent 集中到 Generator，Agent 退化为统一的验证执行器**。

另外，通过引入 **Skill 机制**，将多个原始 MCP 工具封装为高阶能力，Generator 在 Skill 粒度上制定验证计划，进一步降低 token 消耗和出错率。Skill 支持**三级渐进加载**，按上下文相关性按需加载，使架构能扩展到 100+ 工具场景。

## 2. 架构全景

### 2.1 两条路径

```
                        ┌─────────────────┐
                        │  Smart Router   │
                        └────────┬────────┘
                                 │
                    ┌────────────┴────────────┐
                    │                         │
               不需要排查                  需要排查/操作
               (FAQ/咨询)                (不管简单还是复杂)
                    │                         │
                    ▼                         ▼
           ┌────────────────┐     ┌────────────────────┐
           │  RAG 直答       │     │  hypothesis_graph   │
           │  1次LLM, 秒回  │     │                     │
           └────────────────┘     │  Generator 控制复杂度│
                                  │  简单→1个假设        │
                                  │  复杂→3个假设        │
                                  └────────────────────┘
```

只有两条路径：

1. **RAG 直答**：FAQ/知识咨询，无需排查，RAG 检索 + 1 次 LLM 生成回答
2. **Hypothesis Graph**：所有需要排查/操作的场景，Generator 通过假设数量自动调节复杂度

**假设数量是复杂度的旋钮，不需要不同的代码路径来区分简单和复杂问题。**

## 3. 组件详细设计

### 3.1 Smart Router（意图分类器）

**职责**：判断用户请求走 RAG 直答还是 hypothesis_graph。

**分类信号**：

| 信号 | direct_answer | hypothesis_graph |
|------|--------------|-----------------|
| 动作词（查/修/重启/回滚/扩容） | 无 | 有 |
| 症状词（挂了/报错/超时/502/OOM） | 无 | 有 |
| 咨询词（是什么/怎么/推荐/流程） | 有 | 可有可无 |
| RAG 检索置信度 | > 0.8 | 不关心 |

**分类逻辑**（伪代码）：

```python
def classify(request, rag_results) -> Literal["direct_answer", "hypothesis_graph"]:
    has_action_words  = match(request.message, ACTION_KEYWORDS)
    has_symptom_words = match(request.message, SYMPTOM_KEYWORDS)
    rag_confidence    = rag_results.top_score

    # 无排查信号 + RAG 高置信命中 → 直答
    if not has_action_words and not has_symptom_words and rag_confidence > 0.8:
        return "direct_answer"

    # 其他一律走假设验证
    return "hypothesis_graph"
```

**实现方式**：规则优先，可选 LLM 兜底。Router 本身是轻量级的，不做复杂推理。

### 3.2 RAG 直答路径

**适用场景**：FAQ 类知识咨询，如"K8s HPA 是什么"、"发布流程是什么"。

**流程**：

1. RAG Service 检索（sparse + dense + rerank）
2. 1 次 LLM 调用，基于检索结果生成回答
3. 直接返回，无 Agent、无工具调用、无审批

**成本**：~1.5K tokens，<1s 响应。

### 3.3 Context Collector（上下文采集节点）

**职责**：围绕故障时间窗口，聚合多源上下文，构建故障现场快照。同时负责 **Skill 分类过滤**，决定哪些 Skill 类别与当前上下文相关。

**输入**：用户请求（message, service, cluster 等）

**执行步骤**：

1. **RAG 知识检索**：调用 RAG Service，获取相关知识片段
2. **历史案例查询**：从 IncidentCaseStore 查询相似故障（按 service + 症状关键词）
3. **MCP 工具采集**（可选）：如果请求中包含 service/cluster，主动拉取当前状态指标
4. **Skill 分类过滤**（Level 0 → Level 1）：根据上下文关键词匹配相关 Skill 分类，加载其签名

**输出**：`ContextSnapshot`

```python
class ContextSnapshot(BaseModel):
    request: TicketRequest                 # 原始请求
    rag_context: RAGContextBundle          # RAG 检索结果
    similar_cases: list[IncidentCase]      # 相似历史案例 (0-5条)
    live_signals: dict[str, Any]           # 实时信号 (可选)
    context_quality: float                 # 上下文质量评分 (0-1)
    available_skills: list[SkillSignature] # 过滤后的 Skill 签名 (Level 1)
```

### 3.4 Skill 层（高阶能力封装 - 新增）

Skill 是整个架构中新增的关键抽象层，位于 Generator / VerificationAgent 和原始 MCP 工具之间。

#### 3.4.1 为什么需要 Skill

原始 MCP 工具粒度太细，Generator 直接操作工具存在三个问题：

```
问题1: Token 浪费
  30 个工具的完整签名 ~600 tokens
  Generator 每次调用都要传入

问题2: Generator 负担重
  "要验证 OOM，我需要调 k8s.get_pod_metrics + k8s.get_events + k8s.get_logs，
   参数分别是..."
  → 容易出错（参数名写错、漏调工具）

问题3: 不可复用
  每个假设的验证计划都要从零写起
  相同的诊断逻辑无法跨假设复用
```

Skill 将多个工具调用 + 分析逻辑封装为一个高阶能力：

```
原始工具:   k8s.get_pods + k8s.get_events + k8s.get_pod_metrics → 手动组合分析
Skill:     check_pod_health(service) → 内部编排 3 个工具 + 综合判断 → 结构化结果
```

#### 3.4.2 Skill 分类

Skill 分为两类：

| 类型 | 说明 | 示例 | 风险标记 |
|------|------|------|---------|
| **诊断 Skill** | 只读操作，采集证据 | check_pod_health, check_memory_trend | `risk: low` |
| **操作 Skill** | 写操作，改变系统状态 | restart_pods, rollback_deploy | `risk: high` |

诊断 Skill 由 VerificationAgent 在验证阶段调用。操作 Skill 由 Execute 阶段在审批通过后调用。两者使用同一套注册和执行机制。

#### 3.4.3 Skill 目录

```
[K8s]
- check_pod_health(service) — Pod 健康状态综合检查 (get_pods + get_events + get_metrics)
- check_memory_trend(service, window) — 内存趋势分析 (get_pod_metrics + 趋势计算)
- check_resource_limits(service) — 资源配额与实际用量对比
- restart_pods(service) — 重启 Pod [高风险]
- scale_replicas(service, count) — 扩缩容 [中风险]

[CICD]
- check_recent_deploys(service) — 最近部署变更检查 (get_deploys + diff)
- check_pipeline_status(project, branch) — 流水线状态
- rollback_deploy(service, version) — 回滚部署 [高风险]

[Network]
- check_network_latency(service, upstream) — 链路延迟分析 (get_latency + trace)
- check_dns_resolution(domain) — DNS 解析检查
- check_ingress_rules(service) — Ingress 配置检查

[DB]
- check_db_health(service) — 数据库连接池 + 慢查询综合检查
- check_replication_lag(instance) — 主从复制延迟

[Monitor]
- check_log_errors(service, window) — 错误日志聚合分析
- check_alert_history(service) — 告警历史查询
- check_slo_status(service) — SLO 达标状态
```

5 个分类，~15 个 Skill，覆盖主要 IT 运维诊断和操作场景。

#### 3.4.4 三级渐进加载

Skill 支持按需加载，避免全量传入 Generator：

```
Level 0 — 分类索引 (~50 tokens)
  Context Collector 阶段使用
  只包含分类名 + 描述 + skill 数量
  用于判断哪些分类与当前上下文相关

  [K8s] Kubernetes 集群操作 (5 skills)
  [CICD] CI/CD 流水线 (3 skills)
  [Network] 网络诊断 (3 skills)
  [DB] 数据库 (2 skills)
  [Monitor] 监控告警 (3 skills)

Level 1 — Skill 签名 (~20 tokens/skill)
  Generator 阶段使用，只加载相关分类
  包含函数签名 + 一句话描述

  - check_pod_health(service) — Pod健康综合检查
  - check_memory_trend(service, window) — 内存趋势分析

Level 2 — Skill 实例 (按需)
  VerificationAgent 执行阶段，只实例化 plan 中用到的 Skill
  包含完整实现代码、内部工具编排逻辑
```

**过滤示例**：

```
用户: "order-service 偶尔重启"

Level 0 过滤:
  ✅ K8s     — "重启" 命中
  ✅ CICD    — 可能是部署导致
  ✅ Monitor — 需要查日志
  ❌ Network — 不涉及
  ❌ DB      — 不涉及

Level 1 加载: 5(K8s) + 3(CICD) + 3(Monitor) = 11 个 Skill 签名 → ~220 tokens

Generator 生成 plan:
  H1: check_pod_health + check_memory_trend          → 用了 2 个 Skill
  H2: check_recent_deploys + check_log_errors        → 用了 2 个 Skill

Level 2 实例化: 只加载这 4 个 Skill，其余 7 个不加载
```

**Token 消耗随 Skill 总量的变化**：

```
                全量加载      渐进加载 (过滤后)     节省

15 个 skills:    ~350t         ~180t               49%
50 个 skills:    ~1200t        ~250t               79%
100 个 skills:   ~2400t        ~300t               87%
```

#### 3.4.5 多轮对话中的动态追加

渐进加载在多轮对话场景下支持动态追加新的 Skill 分类：

```
Round 1:
  用户: "order-service 偶尔重启"
  加载: K8s + CICD + Monitor skills
  结论: "疑似 OOM，但需要进一步确认"

Round 2:
  用户: "会不会是上游调用导致的？"
  追加加载: Network skills              ← 增量追加
  Generator 看到: K8s + CICD + Monitor + Network

  新假设: "上游超时导致连接堆积 → OOM"
    plan:
      - skill: check_network_latency    ← 用新追加的 Skill
      - skill: check_pod_health         ← 复用已有的 Skill
```

#### 3.4.6 Skill 数据模型

```python
class SkillCategory(BaseModel):
    """Level 0: Skill 分类"""
    name: str                              # "k8s"
    description: str                       # "Kubernetes 集群操作"
    skill_count: int                       # 5
    match_keywords: list[str]              # 用于 Context Collector 过滤

class SkillSignature(BaseModel):
    """Level 1: Skill 签名 (传给 Generator)"""
    name: str                              # "check_pod_health"
    params: str                            # "(service)"
    description: str                       # "Pod健康综合检查"
    risk_level: str                        # "low" / "high"
    category: str                          # "k8s"

class Skill(BaseModel):
    """Level 2: Skill 完整定义"""
    name: str
    description: str
    params_schema: dict[str, str]          # {service: str, namespace: str}
    risk_level: str
    tools_used: list[str]                  # 内部用了哪些 MCP 工具 (审计用)

class SkillResult(BaseModel):
    """Skill 执行结果"""
    skill_name: str
    status: str                            # "healthy" / "unhealthy" / "error"
    summary: str                           # 一句话结论
    evidence: list[dict[str, Any]]        # 结构化证据
    raw_tool_results: list[dict]          # 原始工具返回 (审计用)
```

#### 3.4.7 Skill Registry（注册中心）

```python
class SkillRegistry:
    """Skill 注册中心，支持三级渐进加载"""

    def get_categories(self) -> list[SkillCategory]:
        """Level 0: 返回全部分类索引 (~50 tokens)"""
        ...

    def get_signatures(self, categories: list[str]) -> list[SkillSignature]:
        """Level 1: 返回指定分类的 Skill 签名"""
        ...

    def get_skill(self, skill_name: str) -> Skill:
        """Level 2: 返回 Skill 实例 (执行时才调用)"""
        ...

    def register(self, skill_impl: SkillImplementation) -> None:
        """注册新 Skill 实现"""
        ...
```

新增 Skill 的流程：编写 Skill 实现类 → 注册到 SkillRegistry → 自动出现在分类目录中。无需修改 Generator prompt 或其他组件。

#### 3.4.8 Skill 实现示例

```python
class CheckPodHealthSkill:
    """Pod 健康检查 Skill"""
    name = "check_pod_health"
    category = "k8s"
    risk_level = "low"
    tools_used = ["k8s.get_pods", "k8s.get_events", "k8s.get_pod_metrics"]

    async def execute(self, params: dict, mcp: MCPClient) -> SkillResult:
        service = params["service"]

        # 内部编排多个工具调用
        pods = await mcp.call_tool("k8s.get_pods", {"label_selector": f"app={service}"})
        events = await mcp.call_tool("k8s.get_events", {"namespace": "default", "type": "Warning"})
        metrics = await mcp.call_tool("k8s.get_pod_metrics", {"pod_name": f"{service}-*"})

        # 综合分析
        has_oom = any("OOMKilled" in str(e) for e in events)
        has_crash = any("CrashLoopBackOff" in str(p) for p in pods)

        if has_oom:
            return SkillResult(
                skill_name=self.name,
                status="unhealthy",
                summary=f"{service} 存在 OOMKilled 事件",
                evidence=[{"type": "OOMKilled", "pods": pods, "events": events}],
                raw_tool_results=[pods, events, metrics],
            )
        if has_crash:
            return SkillResult(
                skill_name=self.name,
                status="unhealthy",
                summary=f"{service} Pod 处于 CrashLoopBackOff",
                evidence=[{"type": "CrashLoopBackOff", "pods": pods}],
                raw_tool_results=[pods, events, metrics],
            )
        return SkillResult(
            skill_name=self.name,
            status="healthy",
            summary=f"{service} Pod 运行正常",
            evidence=[],
            raw_tool_results=[pods, events, metrics],
        )
```

### 3.5 Hypothesis Generator（假设生成器）

**职责**：基于上下文 + 历史案例 + 可用 Skill 列表，生成 1-3 个根因假设，每个假设附带 Skill 粒度的验证计划。

**核心**：1 次 LLM 调用，是整个系统的"大脑"。

**输入**：`ContextSnapshot`（包含 `available_skills` 字段，由 Context Collector 过滤后传入）

**LLM Prompt 结构**：

```
System: 你是 IT 运维根因分析专家。基于故障上下文和历史案例，
       推测最可能的根因假设，并为每个假设制定验证计划。

Context:
  - 用户描述: {request.message}
  - 服务: {request.service}
  - 集群: {request.cluster}
  - RAG 知识: {rag_context.top_chunks}
  - 相似历史案例:
    案例1: {case.symptom} → 根因: {case.root_cause} (已验证)
    案例2: ...

可用 Skills:
  [K8s]
  - check_pod_health(service) — Pod健康综合检查
  - check_memory_trend(service, window) — 内存趋势分析
  - check_resource_limits(service) — 资源配额检查
  [CICD]
  - check_recent_deploys(service) — 最近部署变更检查
  - check_pipeline_status(project, branch) — 流水线状态
  [Monitor]
  - check_log_errors(service, window) — 错误日志聚合
  - check_alert_history(service) — 告警历史

Instructions:
  1. 根据上下文复杂度，生成 1-3 个假设
  2. 每个假设的验证计划使用上方列出的 Skill（不要使用未列出的 Skill）
  3. 每个假设必须包含:
     - root_cause: 假设的根因描述
     - confidence_prior: 先验置信度 (基于上下文和案例匹配)
     - verification_plan: Skill 调用列表，每步指定 skill 名和参数
     - expected_evidence: 如果假设成立，预期看到什么证据
  4. 假设数量规则:
     - 上下文明确、历史案例高度匹配 → 1 个假设
     - 有 2-3 种可能方向 → 2-3 个假设
     - 最多 3 个，避免过度发散

Output: JSON list of Hypothesis objects
```

**输出**：`list[Hypothesis]`

```python
class VerificationStep(BaseModel):
    skill_name: str                    # Skill 名，如 "check_pod_health"
    params: dict[str, Any]             # Skill 参数
    purpose: str                       # 这步要验证什么

class Hypothesis(BaseModel):
    hypothesis_id: str                 # "H1", "H2", "H3"
    root_cause: str                    # 假设的根因
    confidence_prior: float            # 先验置信度 (0-1)
    verification_plan: list[VerificationStep]  # Skill 粒度的验证计划
    expected_evidence: str             # 预期证据描述
    recommended_action: str            # 如果确认，建议执行的操作 Skill
    action_risk: str                   # 操作风险等级: low / high
    action_params: dict[str, Any]      # 操作参数
```

**假设数量的自动控制**：

Generator 的 LLM prompt 中包含规则，让模型根据上下文自行判断生成几个假设。不需要外部逻辑控制——这是 LLM 擅长的判断。

**Generator 与 Skill 的关系**：

Generator 只看到 Skill 签名（Level 1），不看到原始工具。这带来三个好处：
1. **Token 更省**：10-15 个 Skill 签名 ~250 tokens，vs 30 个工具签名 ~600 tokens
2. **出错率更低**：Generator 不需要记住每个工具的参数细节，Skill 内部处理
3. **验证计划更短**：2-3 步 Skill 调用 vs 6-9 步工具调用

### 3.6 VerificationAgent（统一验证 Agent）

**职责**：接收 1 个假设 + Skill 粒度的验证计划，按步骤调用 Skill，评估证据，返回置信度。

**与领域 Agent 的本质区别**：

```
领域 Agent (删除):
  角色 = 领域专家 (推理 + 执行)
  输入 = "这个问题交给你"
  过程 = Agent 自己决定查什么工具
  知识 = 硬编码在 Agent 代码和 prompt 中

VerificationAgent (新增):
  角色 = 通用执行器 (只执行)
  输入 = "验证这个假设, 按这个计划调 Skill"
  过程 = 严格按 verification_plan 逐步调用 Skill
  知识 = 不需要领域知识, 领域知识在 Generator + Skill 内部
```

**关键特性：VerificationAgent 不需要加载工具描述，也不需要 Skill 的完整定义。** 它只需要按 plan 中的 skill_name 调用 SkillRegistry，由 Skill 内部编排 MCP 工具。这意味着 VerificationAgent 的 token 消耗极低。

**实现**：

```python
class VerificationAgent:
    """统一的假设验证 Agent，通过 SkillRegistry 调用 Skill。"""

    def __init__(self, skill_registry: SkillRegistry):
        self.skill_registry = skill_registry

    async def verify(self, hypothesis: Hypothesis, context: ContextSnapshot) -> VerificationResult:
        evidence_items = []

        # 1. 按计划逐步调用 Skill (Level 2 按需加载)
        for step in hypothesis.verification_plan:
            skill = self.skill_registry.get_skill(step.skill_name)
            skill_result = await skill.execute(step.params)
            evidence_items.append(EvidenceItem(
                skill=step.skill_name,
                purpose=step.purpose,
                result=skill_result.model_dump(),
                matches_expected=self._matches(skill_result, hypothesis.expected_evidence),
            ))

        # 2. 评估证据与预期的匹配度
        evidence_strength = self._evaluate_evidence(
            evidence_items, hypothesis.expected_evidence
        )

        # 3. 计算最终置信度
        confidence = self._calculate_confidence(
            prior=hypothesis.confidence_prior,
            evidence_strength=evidence_strength,
        )

        return VerificationResult(
            hypothesis_id=hypothesis.hypothesis_id,
            root_cause=hypothesis.root_cause,
            confidence=confidence,
            evidence=evidence_items,
            evidence_strength=evidence_strength,
            recommended_action=hypothesis.recommended_action,
            action_risk=hypothesis.action_risk,
            action_params=hypothesis.action_params,
        )
```

**证据评估**：VerificationAgent 内部用轻量 LLM 调用（~0.5K tokens）对比实际证据和预期证据的匹配程度。也可以用规则匹配（如检查 SkillResult.status 是否为 "unhealthy"）。

### 3.6 Parallel Verification（并行验证调度）

**复用现有 `ParallelDispatcher`**，为每个假设创建一个 VerificationAgent 实例并行执行。

```
输入: list[Hypothesis] (1-3 个)
  ↓
ParallelDispatcher.dispatch()
  ├── Semaphore(max_concurrency=3)
  ├── 为每个假设创建 HypothesisTask
  ├── asyncio.gather() 并行执行
  └── 收集 VerificationResult 列表
  ↓
输出: list[VerificationResult]
```

当 N=1 时，实质上退化为串行执行单个验证，无额外开销。

### 3.7 Hypothesis Ranker（排序器）

**职责**：按置信度对验证结果排序，输出推荐根因。

**打分公式**：

```
final_score = w1 * evidence_strength    # 证据强度 (VerificationAgent 返回)
            + w2 * confidence            # Agent 置信度
            + w3 * history_match         # 历史案例匹配度 (来自 ContextSnapshot)
```

默认权重：`w1=0.5, w2=0.3, w3=0.2`。权重由 Feedback Gate 根据历史准确率自适应调整。

**输出**：

```python
class RankedResult(BaseModel):
    primary: VerificationResult           # 推荐根因（最高分）
    secondary: list[VerificationResult]   # 补充/关联根因
    rejected: list[VerificationResult]    # 证据不足，排除
    ranking_metadata: dict[str, Any]      # 打分详情

    # 分类阈值
    # primary: score 最高且 > 0.6
    # secondary: score > 0.4 但非最高
    # rejected: score <= 0.4
```

**N=1 时的行为**：只有一个假设，跳过排序，直接作为 primary 输出。

### 3.8 Approval Gate（审批门）

**复用现有实现**，从 `RankedResult.primary.recommended_action` 提取操作提案。

流程不变：
- 低风险操作 → auto-approve
- 高风险操作 → 推送人工审批（InterruptStore）
- 拒绝 → 终止

### 3.9 Execute（执行层）

**复用现有实现**：`ExecutionManager` + `ExecutionSafetyValidator`。

- Precheck → Primary Action (MCP 工具调用) → Record
- 重试策略 + 补偿策略
- CheckpointStore 恢复点

### 3.10 Feedback Gate（反馈门 - 新增）

**职责**：人工确认诊断结果，写入案例库，更新 Ranker 权重。

这是与现有架构最大的新增点，实现**学习闭环**。

**流程**：

  ┌──────────────────────────────────────┐

  │  ✅ 根因正确                          │

确认正确 → 结构化写入 IncidentCaseStore，human_verified = true

## 6. 成本对比

### 6.1 Token 消耗

```
场景: 复杂故障排查 (3 个方向)

现有分工模式:
  Supervisor 路由 LLM               ~1.0K tokens
  CICDAgent.diagnose()              ~2.5K tokens  (含工具 schema ~0.8K)
  NetworkAgent.diagnose()           ~2.5K tokens  (含工具 schema ~0.8K)
  SREAgent.diagnose()               ~2.5K tokens  (含工具 schema ~1.2K)
  ──────────────────────────────
  总计                              ~8.5K tokens  (其中工具描述 ~2.8K)

新假设竞争模式 (含 Skill + 渐进加载):
  Context Collector                  0K (工具调用 + Skill 过滤)
  Hypothesis Generator              ~3.5K tokens  (含过滤后 Skill 签名 ~0.2K)
  VerificationAgent × 3             ~1.5K tokens  (3 × 0.5K 轻量评估)
  ──────────────────────────────
  总计                              ~5.0K tokens  (其中 Skill 描述 ~0.2K)

节省: ~41%
Skill + 渐进加载额外节省: 工具描述从 ~2.8K → ~0.2K
```

### 6.2 效果对比

| 维度 | 分工模式 | 竞争模式 |
|------|---------|---------|
| 模糊问题准确率 | 低（路由可能选错领域） | 高（全局推理 + 多假设覆盖） |
| 明确问题效率 | 高（直达目标 Agent） | 同等（Generator 生成 1 个假设） |
| 跨领域问题 | 差（每个 Agent 视角局限） | 好（Generator 有全局视角） |
| 历史重复故障 | 每次从头诊断 | 案例库命中，高先验置信度 |
| 学习能力 | 无 | 有（Feedback → 案例库 → 权重更新） |

## 7. 实现计划

### Phase 1: 基础管线 (核心路径)

新建 hypothesis_graph 的最小可用版本：

1. **Smart Router**：规则分类（不含 LLM 兜底）
2. **Context Collector**：RAG 检索 + 案例库查询
3. **Hypothesis Generator**：LLM 生成假设 + 验证计划
4. **VerificationAgent**：按计划执行 MCP 工具 + 置信度评估
5. **Ranker**：加权公式排序
6. **Finalize**：生成响应

此阶段跳过 Approval Gate / Execute / Feedback Gate，只做诊断不做执行。

### Phase 2: 执行与审批

1. 对接现有 **Approval Gate**（复用 ApprovalCoordinator）
2. 对接现有 **Execute**（复用 ExecutionManager）
3. 对接现有 **Interrupt 机制**（审批中断/恢复）

### Phase 3: 反馈闭环

1. **Feedback Gate**：人工确认/纠正中断
2. **案例库增强**：IncidentCase 扩展字段
3. **权重自适应**：基于历史准确率更新 Ranker 权重

### Phase 4: 清理与迁移

1. 删除旧 ticket_graph 及相关代码
2. 删除 RuleBasedSupervisor
3. 删除所有领域 Agent 类及配置
4. 删除 Aggregator
5. 更新 API 层适配新 graph

## 9. 关键设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 两条路径（RAG直答 + hypothesis_graph）而非三条 | 假设数量是复杂度旋钮，不需要额外代码路径 |
| D2 | 统一 VerificationAgent 取代多领域 Agent | 竞争模式下 Agent 只做执行不做推理，领域知识在 Generator + Skill |
| D3 | Hypothesis Generator 用 LLM + 案例库融合 | 纯 LLM 缺乏领域沉淀，纯案例库无法处理新故障 |
| D4 | Ranker 用加权公式而非 LLM | 省 token，可解释性好，权重可自适应调整 |
| D5 | Feedback Gate 作为可选节点 | 不阻塞主流程，但提供学习闭环能力 |
| D6 | 分 4 个 Phase 实现 | Phase 1 最小可用，逐步叠加，降低风险 |
| D7 | 引入 Skill 层封装 MCP 工具 | Generator 在 Skill 粒度制定计划，降低 token 消耗和出错率；Skill 可独立测试和复用 |
| D8 | Skill 三级渐进加载 | 按上下文相关性按需加载，使架构扩展到 100+ 工具场景；多轮对话支持动态追加 |
| D9 | 诊断 Skill 与操作 Skill 统一机制 | 同一套 SkillRegistry 管理，操作 Skill 标记 risk_level，经审批后才执行 |
